
Introducción a la configuración avanzada

La configuración inicial del protocolo de red NetWare® 4.1 proporciona la base para la mayor parte de las necesidades.
La configuración se basa en las tarjetas configuradas y en los protocolos habilitados y configurados.

Si siguió las instrucciones de la Guía básica de configuración de protocolo NetWare 4.1 , durante la configuración inicial, la mayor parte de los parámetros del protocolo están definidos con sus valores por defecto.
Esta configuración básica de servidor es útil para casi todos los propósitos de una LAN.

El software de protocolo NetWare 4.1 ofrece un extenso conjunto de parámetros configurables para todos los protocolos soportados.
Puede hacer uso de estos parámetros para modificar las características por defecto de la configuración original.
Puede configurar las características avanzadas y optimizar las opciones de rendimiento del protocolo.
En esta guía se presentan conceptos y procedimientos avanzados de configuración, y discute las utilidades de gestión que podría encontrar útiles en su situación particular.

Introducción a la configuración

Para configurar un servidor para su funcionamiento en una LAN, debe realizar las siguientes tareas usando INETCFG: 
- Configurar la tarjeta LAN.
Para una tarjeta LAN, sólo es necesario configurar la tarjeta.
Para obtener información acerca de la configuración de tarjetas LAN, consulte la Guía básica de configuración de protocolos NetWare 4.1.

- Configurar los protocolos de red que se ejecutarán sobre la conexión LAN.
Para una LAN, puede configurar el protocolo de encaminamiento (para IPX y TCP/IP), túnel (para IPX y AppleTalk*), filtro, rutas estáticas (sólo para TCP/IP) y otras configuraciones expertas.
Para obtener información acerca de la configuración de protocolos de red para una LAN, consulte el capítulo 2, "Configuración de IPX," capítulo 3, "Configuración de IP," y el capítulo 4, "Configuración de AppleTalk." 

- Asocie los protocolos a las interfaces de red a través de las cuales desee que se conecten.
Para obtener información acerca de cómo asociar los protocolos, consulte el capítulo 2, "Configuración de IPX," capítulo 3, "Configuración de IP," y el capítulo 6, "Configuración de AppleTalk,"


Conceptos básicos de configuración

La configuración del servidor implica cierta planificación y toma de decisiones sobre la marcha.
Debe conocer los detalles de la red en la que está situando el servidor, el tipo de conectividad que desea y los detalles de configuración de los pares a los que se está conectando el servidor.

La decisión de las características de cada protocolo utilizado sobre una interfaz LAN se basa en la topografía y características existentes de la red.
Para cada protocolo configurado para su ejecución sobre un enlace LAN,debe determinarse si se encamina y qué protocolo utilizar, si se establece el túnel de paquetes, si se necesita o desea configurar rutas estáticas y el tipo de filtro que se ha de usar.

En esta sección se tratan los conceptos básicos que debe conocer para tomar las decisiones correctas al configurar los protocolos de red que desee usar en el software NetWare 4.1.
Cada protocolo de red que vaya a soportarse debe estar asociado y debe ejecutarse sobre un enlace configurado.
Para obtener información acerca de conceptos de configuración de enlaces y configuración de protocolos, consulte el capítulo 2, "Configuración de IPX," capítulo 3, "Configuración de IP," y el capítulo 4, "Configuración de AppleTalk,".

Protocolos de encaminamiento

Tanto la configuración de IPX como la de TCP/IP ofrecen opciones para establecer los protocolos de encaminamiento.
Hay dos tipos generales de protocolos de encaminamiento disponibles: vector de distancia y estado del enlace.
La elección del protocolo de encaminamiento se determina en gran medida por los protocolos actualmente utilizados en la red.
Por ejemplo, en redes más antiguas, como NetWare 2, podría haber varios routers capaces de soportar únicamente encaminamiento por vector de distancias.
Sin embargo, si su situación le permite elegir sólo un tipo de encaminamiento, seleccione si es posible el de estado del enlace.

Si su red contiene routers que soportan tanto encaminamiento por estado del enlace como por vector de distancia, deberá configurar su router para soportar ambos.
En redes que sólo contienen los routers más modernos, lo mejor es configurarlos para que utilicen sólo protocolos de encaminamiento por estado del enlace.
Aunque no es un escenario muy probable, sería posible alcanzarlo actualizando gradualmente los routers y migrando al uso exclusivo de protocolos de estado del enlace en un período de tiempo.

Enlaces permanentes y rutas estáticas

Puede hacer uso de rutas estáticas para conexiones LAN en TCP/IP.
Estas rutas estáticas se utilizan cuando se desea eliminar tráfico de protocolo de encaminamiento en la conexión, aumentando el ancho de banda disponible para los datos, o para tener una ruta de respaldo de las rutas dinámicas.
También puede utilizar rutas estáticas para limitar el acceso de los usuarios a una parte de la red, o bien para obtener acceso a partes aisladas de la misma.

Túnel

El software NetWare 4.1 proporciona el controlador IPTUNNEL para el túnel de paquetes IPX a través de las interredes IP, y el protocolo AURP para el túnel de paquetes AppleTalk a través de interredes IP.

IPTUNNEL está configurado como un controlador de tarjeta y utiliza encapsulado UDP para enviar paquetes IPX a través de una interred IP a un par remoto IPTUNNEL.
El par remoto IPTUNNEL deshace el encapsulado UDP y envía el paquete IPX a través de una red IPX a su destino.

Análogamente, AURP encapsula paquetes AppleTalk para enviarlos a través de una interred IP a un par AURP remoto.
Éste a su vez deshace el encapsulado y envía el paquete a través de la red AppleTalk hasta su destino.

Filtros

Los protocolos NetWare 4.1 soportan filtros para controlar la información de servicio y encaminamiento aceptada o notificada por un servidor.
Los filtros son útiles cuando desea limitar tipos específicos de tráfico a determinadas partes de su red y cuando desea ofrecer cierto nivel de seguridad.
La Filter Configuration Utility (Utilidad de Configuración de Filtros) (FILTCFG) le permite configurar los filtros para IPX, IP y AppleTalk para descartar de forma selectiva paquetes enviados o recibidos por un servidor.
Se soportan los siguientes tipos de filtros:
- Reenvío de paquetes.
Para evitar que el servidor reenvíe ciertos paquetes de datos.
El filtro de reenvío de paquetes está disponible en IPX y TCP/IP.

- Información de servicios.
Para evitar que el servidor notifique paquetes de notificación de servicios destinados a zonas particulares de la red.
El filtro de información de servicios está disponible en IPX y AppleTalk.

- Información de encaminamiento.
Para limitar las rutas añadidas a la tabla de rutas de servidores especificados.
El filtro de información de encaminamiento está disponible en IPX, AppleTalk y TCP/IP.


Conceptos básicos de gestión

El software de red NetWare 4.1 ofrece diversas utilidades para gestionar el servidor y su operación en la interred.
La gestión del servidor después de su configuración consta de las siguientes tareas principales:
- Verificar la aplicación para asegurarse de que todo está activo y en ejecución y que tiene las características deseadas.
- Monitorizar el servidor para averiguar si está funcionando de forma eficiente y para detectar posibles problemas.
- Configurar y gestionar equipos de conexión
- Optimizar el servidor para hacer más eficiente el encaminamiento
- Pruebas de conectividad y resolución de problemas

Entre las utilidades disponibles para ayudarle a configurar y gestionar su servidor se incluyen:
- Utilidades de gestión y pruebas, tales como INETCFG, FILTCFG o IPXPING, para configurar y optimizar el servidor.
- Utilidades de monitorización, tales como IPXCON, ATCON o TCPCON, para ayudarle a comprobar cómo se está comportando cada protocolo e identificar problemas potenciales.
- Utilidades de línea de comandos, tales como PING, CONLOG o REINITIALIZE SYSTEM, para ayudarle monitorizar, probar e implementar cambios de configuración en su servidor.

Las características de configuración avanzadas descritas en esta guía incluyen información tal como la configuración de parámetros de bajo nivel de enlace de datos, incluyendo límites de colas de transmisión y recuperación de errores de protocolo; y técnicas de optimización para protocolos de redes y enlaces LAN.

Aunque pueden utilizarse muchas técnicas de optimización en una LAN, dicha optimización no es esencial ya que el ancho de banda no es caro.

La monitorización de su servidor durante y después de la configuración inicial, puede ayudarle a detectar problemas potenciales e identificar sus necesidades de optimización.
También puede hacer uso conjunto de las utilidades de configuración para aislar problemas.
Para obtener información acerca de las utilidades, consulte el capítulo 6, "Configuración de AppleTalk".
Para informarse acerca del la resolución de problemas, consulte el capítulo 9, "Descripción general de la resolución de problemas."

Decisiones sobre la configuración de IPX

La configuración de IPX, aparte de la más básica, depende de las siguientes decisiones:
- Si va a desactivarse el reenvío de paquetes IPX o si se va a utilizar el sistema como un router IPX NetWare 4.1 permite desactivar, o inhabilitar, el reenvío de paquetes IPX en un servidor de archivos NetWare.
Esto es útil cuando se dispone de más de un sistema NetWare conectando dos o más LANs y se desea liberar a uno de los sistemas de la tarea de reenviar paquetes IPX entre las LANs.
Para desactivar el reenvío de paquetes IPX, consulte "Desactivación del reenvío de paquetes IPX" en la página 10.

- Si va a utilizarse NLSP, RIP/SAP o ambos en el servidor.
NetWare Link Services Protocol (Protocolo de servicios de enlace NetWare) (NLSP) es el protocolo Novell® de encaminamiento por estado de enlace para interredes IPX; RIP y SAP son los protocolos de encaminamiento y notificación de servicios usados tradicionalmente por NetWare.
Para configurar NLSP en su servidor, consulte "Configuración de NLSP" en la página 12.
Para configurar RIP y SAP en su servidor, consulte "Configuración de RIP y SAP" en la página 19.

- Si van a habilitarse los parámetros avanzados de IPX. Se trata de un conjunto de parámetros con los cuales se configura NLSP y otras características avanzadas del protocolo IPX.
Para habilitar estos parámetros para configuración, consulte "Habilitación de parámetros avanzados de IPX" en la página 25.

- Si van a habilitarse los filtros de ruta, servicio o paquetes IPX o NetBIOS configurados mediante la Filter Configuration Utility (Utilidad de Configuración de Filtros) (FILTCFG). Los filtros permiten hacer lo siguiente:
- Controlar el acceso de usuarios a cualquier servicio.
- Reducir el ancho de banda consumido por el tráfico de encaminamiento
- Restringir el flujo de paquetes IPX o NetBIOS.
Para configurar filtros, debe habilitar primero el parámetro Filtering Support desde INETCFG y luego cargar la Filter Configuration Utility (Utilidad de Configuración de Filtros) (FILTCFG).
Para obtener más información, consulte el capítulo 5, "Configuración de filtros." 


- Si va a modificarse la forma en la que el router propaga paquetes de tipo 20.
El tipo 20 es una clase de paquete IPX que se refiere a cualquier paquete propagado.
Por ejemplo, los paquetes NetBIOS son tipo 20.
Si su router procesa gran cantidad de paquetes tipo 20, puede controlar cómo propaga dichos paquetes a través de sus interfaces.
Esto puede reducir el tráfico en una LAN.
Para cambiar la forma en que el router propaga paquetes de tipo 20, consulte "Control de la propagación de paquetes de tipo 20" en la página 27.

- Si va a cambiarse el límite del contador de saltos de paquetes IPX salientes.
Esto permite controlar el rango de paquetes salientes en su servidor.
Para controlar el rango de paquetes IPX, consulte "Cambio del límite de saltos para paquetes IPX" en la página 29.

- Si va a hacerse balance de tráfico sobre rutas de igual coste.
Si su router soporta dos o más interfaces de red que pueden alcanzar el mismo destino, puede distribuir el tráfico saliente entre dichas interfaces para aumentar el rendimiento.
Para configurar rutas de igual coste, consulte "Balance de cargas de tráfico entre rutas de igual coste" en la página 31.

- Si van a configurarse parámetros de conexión SPX.
Podría necesitar configurar ciertos parámetros de Sequenced Packet Exchange (Intercambio de Paquetes Secuenciados ) (SPX ) si está ejecutando aplicaciones NetWare que tienen requisitos específicos para tiempo límite de conexión, total de reintentos, etc. 
Para configurar cualquiera de estos parámetros, consulte "Configuración de parámetros de conexión SPX" en la página 33.

- Si va a configurarse el túnel IP.
El túnel IP entre routers pares se utiliza cuando se desea que dos o más redes IPX intercambien información a través de una red de Protocolo Internet (IP).
Para configurar el túnel IP, consulte "Túnel a través de redes IP" en la página 34.

- Si va a configurarse MacIPX gateway. 
MacIPX gateway se utiliza cuando se dispone de clientes Macintosh* en una red AppleTalk* que necesita el uso de recursos disponibles en una LAN NetWare.
Para configurar MacIPX gateway, consulte "Configuración de la Gateway MacIPX" en la página 40.


Desactivación del reenvío de paquetes IPX

Como parte típica de su funcionamiento, un servidor de archivos NetWare reenvía paquetes IPX entre sus interfaces de red.
Inhabilitando el parámetro Packet Forwarding , se desactiva el reenvío de paquetes IPX en un servidor de archivos NetWare.
Esto es útil cuando se dispone de más de un sistema NetWare y se desea disponer de más recursos informáticos para servicios de archivo e impresión.

Un servidor con el reenvío de paquetes IPX desactivado aún realiza sus funciones como servidor de archivos, pero difunde sólo sus propios servicios y número interno de red, y no aquéllos asociados con sus interfaces de red.
Un servidor que opera de esta forma se denomina a veces servidor multiresidente .
Aunque un servidor de este tipo no contesta a peticiones de encaminamiento desde estaciones de trabajo NetWare, debe aún aceptar todas las difusiones RIP y SAP para mantener su bindery .

En la figura 2-1 se muestra cómo pueden dividirse las tareas de encaminamiento y servicio de archivos entre un servidor de archivos NetWare y un router dedicado.
Normalmente, la tarea de encaminar el tráfico IPX entre las dos interredes se compartiría entre el router y el servidor de archivos.
Cuando se inhabilita el reenvío de paquetes IPX en el servidor de archivos, el router asume la tarea de encaminar todo el tráfico IPX.
El servidor de archivos, ahora libre de la tarea de reenviar paquetes IPX y de contestar a peticiones de encaminamiento, puede dedicar todos sus recursos de proceso a servicios de archivos.

Figura

Desactivación del reenvío de paquetes IPX en un servidor de archivos NetWare

Debería desactivar el reenvío de paquetes IPX sólo por las siguientes razones:
- No desea que el servidor reenvíe paquetes IPX entre sus interfaces.

- El servidor opera en una red sólo RIP.
Al desactivar el encaminamiento IPX en una red que posee routers NLSP, se fuerza a que éstos envíen actualizaciones RIP/SAP al servidor.


Método para desactivar el reenvío de paquetes IPX

Para desactivar el reenvío de paquetes IPX, cargue INETCFG y seleccione la siguiente vía de acceso:

Vía de acceso del parámetro: Seleccione Protocols > Seleccione IPX ( aparece la ventana IPX Protocol Configuration )

El parámetro Packet Forwarding estará ya resaltado; selecciónelo y luego seleccione Disabled .

Pulse para volver al menú Internetworking Configuration ; guarde su cambio cuando el sistema lo indique.

Si desea que este cambio tenga efecto inmediato, salga de INETCFG y luego escriba el siguiente comando:

REINITIALIZE SYSTEM >Intro> 

Si desea configurar otros parámetros IPX, hágalo ahora y reinicialice el sistema cuando haya finalizado.

Configuración de NLSP

Novell ha desarrollado NLSP para satisfacer las demandas de grandes interredes IPX.
Como protocolo de encaminamiento por estado del enlace, NLSP ofrece mejor rendimiento, seguridad y capacidad de adaptación de dimensiones que el encaminamiento IPX RIP utilizado tradicionalmente por los servidores NetWare.

A diferencia de RIP y SAP, que difunden periódicamente información de encaminamiento y servicios, NLSP transmite información de encaminamiento sólo cuando tiene lugar un cambio en una ruta o servicio en algún lugar de la red, o bien cada dos horas (lo que ocurra primero).
Como NLSP genera menos actualizaciones de encaminamiento que RIP y SAP, hace uso de un menor ancho de banda de la red para mantener su base de datos de rutas.

Un router NLSP transmite información acerca de los routers que tiene directamente conectados y de los enlaces a éstos en paquetes de estado de enlace (LSPs).
Por defecto, los LSPs son de 512 bytes, un valor nominal suficiente para la mayoría de las redes IPX.
Si su red es grande, del orden de 4.000 rutas y 2.000 servicios o más, debería aumentar el valor del parámetro LSP Size a 1024.
Para configurar este parámetro, consulte "Cambio del tamaño de LSP" en la página 18 .

Por defecto, NLSP difunde sus paquetes porque algunos controladores LAN no soportan de forma adecuada multidifusión , un modo de transmisión que habilita para aceptar un paquete sólo a aquellos dispositivos que tienen su atención en una dirección de paquete multidifusión específica.
Sin embargo, puede cambiar el modo de transmisión de paquetes NLSP a multidifusión mediante el parámetro MAC Channel .
Una ventaja del uso de transmisión por multidifusión es que los paquetes NLSP enviados de esta manera no "obstruyen" con tráfico innecesario nodos que no realizan encaminamiento.

Todos los sistemas NetWare de la misma LAN deben utilizar el mismo modo de transmisión de paquetes NLSP.

NLSP facilita la gestión de grandes interredes IPX permitiendo su partición en dominios administrativos denominados áreas de encaminamiento .
Cada área de encaminamiento puede identificarse por hasta tres direcciones de área, un número único hexadecimal de cuatro bytes que identifica cada router NLSP como parte de un área de encaminamiento.
Aunque las direcciones de área no son necesarias, están disponibles principalmente por compatibilidad con versiones futuras de NLSP y aportan algunas ventajas en grandes interredes IPX.

No configure direcciones de área a menos que deba dividir una gran interred IPX.
Si comete un error en las asignaciones de direcciones, puede seccionar inadvertidamente su red y perder conectividad entre algunos routers .

Para obtener más información acerca de áreas de encaminamiento y división de una interred IPX, consulte la NetWare 4.1 NLSP Migration Guide .

Cada router NLSP se identifica por medio de un número exclusivo hexadecimal de seis bytes denominado ID de sistema.
El ID de sistema por defecto consta de una constante de dos bytes, 0x0200, seguida del propio número interno de red del router.
No debe cambiar el ID de sistema a menos que tenga otro esquema de numeración con el que pueda realizar de forma segura el seguimiento y gestión de los routers NLSP de su red.
Si debe cambiar el ID de sistema de un router, utilice otro número exclusivo, tal como la dirección física de una de sus tarjetas de interfaz de red.

NLSP utiliza un valor de coste por defecto basado en el rendimiento medio para seleccionar la mejor ruta para el envío de paquetes IPX.
NetWare 4.1 permite ignorar este valor en una interfaz.
De esta manera, es posible establecer las rutas preferidas, equilibrar cargas de tráfico entre interfaces y establecer flujos de tráfico específicos entre routers .
Para obtener más información, consulte "Balance de cargas de tráfico entre rutas de igual coste" en la página 31.

Todos los routers NLSP tienen un parámetro configurable denominado Priority .
El router con prioridad más alta se convierte en el router designado , que asume la responsabilidad del intercambio de información de estado del enlace en nombre de todos los demás routers NLSP de la LAN.
Normalmente, no es necesario cambiar el valor del parámetro Priority de un router ; los routers NLSP "eligen" de forma automática uno de entre ellos.
No obstante, si desea forzar la designación de uno de los routers para su LAN, incremente el valor de su Priority por lo menos hasta 85.
El router seleccionado debe estar normalmente activado y tener suficiente memoria para procesar información de encaminamiento NLSP y generar el seudonodo LSP para su LAN.
Si desea evitar que un router particular sea el designado, disminuya su valor de Priority .

NetWare 4.1 proporciona un conjunto de parámetros de convergencia que permiten controlar el funcionamiento de NLSP en su servidor/router.
Los valores por defecto para esos parámetros son suficientes para la mayoría de las redes IPX y deberían cambiarse sólo por consejo de su representante de soporte técnico.

Cuando se configura NLSP en una interfaz, puede establecer el parámetro NLSP State en uno de dos posibles estados: ON y Off.
On habilita al router para intercambiar paquetes NLSP libremente con otros routers NLSP en la red asociada.
Off inhabilita el encaminamiento NLSP en la interfaz.

Para obtener información adicional acerca del funcionamiento de NLSP, consulte NetWare 4.1 IPX Reference.

Configuración de NLSP

Si planea implementar NLSP en toda una red IPX, o incluso si sólo necesita configurar NLSP en unos pocos servidores o LANs, consulte NetWare 4.1 NLSP Migration Guide.

Para configurar NLPS en el servidor, cargue INETCFG y seleccione la siguiente vía de acceso:

Vía de acceso del parámetro: Seleccione Protocols 8> Seleccione IPX ( aparece la ventana IPX Protocol Configuration )

Complete los siguientes pasos:

Seleccione el parámetro Routing Protocol y luego seleccione NLSP with RIP/SAP Compatibility .

Si el router soporta diversas interfaces y necesita filtrar rutas y servicios a un lugar remoto a través de una de estas interfaces, seleccione NLSP with RIP/SAP Compatibility y luego inhabilite NLSP y habilite RIP y SAP en dicha interfaz.
Para habilitar RIP y SAP, consulte "Configuración de RIP y SAP" en la página 19.

Si selecciona Expert Configuration Options , puede visualizar los siguientes parámetros expertos NLSP:
- NLSP Local Area Addresses
- Override NLSP System ID 
- NLSP System Identification 
- NLSP Convergence Rate 
- NLSP Convergence Rate Configuration. La mayor parte de las configuraciones de red no requieren cambios en estos parámetros.
Para aprender acerca de su uso, consulte NetWare 4.1 IPX Reference .


Configuración de IP

El software de NetWare® 4.1 ofrece un juego de parámetros configurables con los que puede modificar características operativas del Protocolo Internet (IP) como, por ejemplo, la selección de su protocolo y su configuración para ejecutarlo en una conexión LAN.

La configuración del IP para NetWare 4.1 se lleva a cabo habilitando el protocolo, definiendo sus diversos parámetros y asociándolo a una interfaz de red.
Todos los parámetros del IP se configuran desde la utilidad Internetworking Configuration o INETCFG (Configuración de interred).

La configuración especificada con INETCFG no surte efecto automáticamente.
Para activarla, salga de INETCFG y, a continuación, desactive el servidor y reinícielo tras finalizar todos los pasos de configuración.

En este capítulo se explican procedimientos de configuración y conceptos avanzados de IP en las secciones siguientes:
- "Decisiones de configuración IP" en la página 51.

Decisiones de configuración IP

El método para llevar a cabo una configuración avanzada de IP depende de las siguientes decisiones:
- Si el computador va a utilizarse como router o como nodo final (es decir, host ). El parámetro IP Packet Forwarding, que controla el encaminamiento de paquetes IP, está inhabilitado por defecto.
Habilítelo si desea que el computador funcione como un router IP; inhabilítelo para que funcione como un nodo final.

- Si desea utilizar un entorno RIP, OSPF o un entorno combinado RIP-OSPF.
RIP y OSPF son protocolos de encaminamiento IP.
Si ya cuenta con routers IP en el entorno de red, utilice el mismo protocolo de encaminamiento que usen dichos routers .
Si la red no tiene otros routers , utilice OSPF.
Para configurar el servidor como router RIP, consulte "Configuración de RIP" en la página 53 .
Para configurar el servidor como router OSPF, consulte "Configuración de OSPF" en la página 57 .

- Si va a utilizar rutas estáticas en un router LAN.
Las rutas estáticas son útiles para reducir el tráfico de encaminamiento, ofrecer seguridad, acceder a redes aisladas y funcionar como rutas de respaldo en las LAN.
Para configurar rutas estáticas en un router LAN, consulte "Configuración de rutas estáticas en routers LAN" en la página 63 .

- Si va a filtrar rutas o varios paquetes TCP/IP Habilite los filtros si desea llevar a cabo algunas de estas acciones:
- Controlar el acceso a cualquier servicio, como el Protocolo de transferencia de archivos (FTP), de la red.
- Reducir el ancho de banda consumido por el tráfico de encaminamiento.
Para configurar los filtros TCP/IP, debe habilitar el parámetro Filtering Support desde INETCFG y cargar a continuación la utilidad Filter Configuration o FILTCFG (Configuración de filtros).
Para obtener más información, consulte el Chapter 5, "Configuración de filtros." 

- Si va a configurar el descubrimiento de router.
Descubrimiento de router permite que los nodos finales localicen un router IP en su red.
Si el computador está funcionando como router, puede auto-notificarse periódicamente como un router; si está funcionando como nodo final, puede enviar consultas para localizar un router.
Para configurar el descubrimiento de router, consulte "Configuración del descubrimiento de router" en la página 66 .

- Si va a habilitar ARP o ARP delegado. ARP es un protocolo LAN que asigna direcciones de Internet a direcciones físicas.
Los nodos finales y routers IP utilizan ARP para determinar la dirección física del nodo de destino.
Un router IRP que utiliza ARP delegado contesta a las peticiones ARP que recibe a través de una interfaz en nombre de un nodo final de una red conectada a otra interfaz.
Para habilitar ARP o configurar ARP delegado en el router, consulte "Habilitación de ARP y ARP delegado" en la página 67 .

- Si va a permitir que el router reenvíe difusiones dirigidas.
Una difusión dirigida es una difusión destinada sólo a un grupo concreto de nodos más que a todos los nodos de la red.
Para habilitar el reenvío de difusión dirigida, consulte "Habilitación del reenvío de difusión dirigida" en la página 69 .

- Si va a configurar el router o nodo final como dispositivo de reenvío BootP. Con el protocolo BootP, los nodos finales obtienen una dirección IP en el momento del arranque.
Como Dispositivo de reenvío BootP , el router o nodo final acepta y reenvía las peticiones BootP a un servidor BootP, que asigna la dirección IP.
Para configurar el reenvío BootP, consulte "Configuración del reenvío BootP" en la página 69 .

- Si va a configurar varias interfaces lógicas en una sola tarjeta.
Esto le permite asociar más de una red IP a una tarjeta LAN para que funcionen como interfaces lógicas separadas.
Para configurar varias interfaces lógicas en una tarjeta, consulte "Configuración de varias interfaces lógicas" en la página 71 .



Configuración de RIP

RIP es, posiblemente, el protocolo de encaminamiento IP de uso más común.
Como tal, su disponibilidad es muy amplia y ofrece pocos obstáculos para la interoperabilidad con otras interredes IP, especialmente con Internet.

RIP funciona bastante bien en interredes IP pequeñas con arquitecturas sencillas y pocos routers .
Sin embargo, RIP revela sus limitaciones en las interredes grandes y complejas que se utilizan cada vez más en las organizaciones gubernamentales y privadas de todo el mundo.
Sus limitaciones más obvias son las siguientes:
- Todas las subredes deben ser contiguas
- Toda la red debe utilizar la misma máscara de subred
- No se puede utilizar subred cero 
- Las rutas RIP se limitan a 15 saltos. 

Para superar o suavizar algunas de estas limitaciones, la comunidad de interredes ha desarrollado varias mejoras para el RIP.
RIP II , por ejemplo, es una versión mejorada de RIP que soporta máscaras de subred de longitud variable por medio de un campo que incluye la máscara de subred de la red de destino.
RIP soporta también el uso de subred cero, cuyas direcciones se reservan bajo la especificación IP original.
Al configurar RIP, puede ejecutar RIP, RIP II o ambos en una sola interfaz.

No todos los routers de terceras partes soportan RIP II.

Puede también habilitar vuelta ficticia en una interfaz.
Se trata de un mecanismo que hace que RIP notifique una ruta de vuelta a través de la misma vía de acceso de la que tomó la ruta, pero con un total de 16 saltos, es decir, inalcanzable.
Aunque la vuelta ficticia evita los bucles de encaminamiento, las rutas inalcanzables efectuadas en cada paquete RIP incrementan el ancho de banda consumido por el tráfico RIP.
Este incremento es muy notable en las interredes grandes.

RIP le permite asignar un valor de coste entre 1 y 5 a cada interfaz de red que configure.
De esta forma puede establecer una ruta preferida de acuerdo con el tipo de medios de red conectados a la interfaz.
Por ejemplo, es posible que desee incrementar el coste de una interfaz que utilice un enlace lento para que, dada la opción, RIP utilice la interfaz con un enlace más rápido y menos costoso.
El coste por defecto de cada interfaz es 1 y no debe incrementarlo a menos que no desee fomentar su uso como vía de encaminamiento.

Al elegir un protocolo de encaminamiento IP, tenga en cuenta las siguientes " reglas empíricas":
- Si la interred IP es pequeña y no utiliza otro protocolo de encaminamiento aparte de RIP, siga utilizando RIP.
Para configurar RIP, consulte "Cómo configurar RIP" en la página 55 .
Sin embargo, si la red sigue creciendo y pasa a formar parte de una interred IP más grande, debe considerar la migración de la red de RIP a OSPF.
Para obtener más información sobre esta migración, consulte la NetWare 4.1 TCP/IP Reference .

- Si la interred utiliza subredes de longitud variable o tiene routers de terceras partes que soportan RIP II, use RIP II o OSPF.
Para configurar RIP II, consulte "Cómo configurar RIP" en la página 55 .
Para configurar OSPF, consulte "Cómo configurar OSPF" en la página 59 .

- Si la interred tiene routers de terceras partes que soportan RIP II y routers que no lo hacen, utilice RIP y RIP II.
Con este propósito, puede habilitar RIP y RIP II de forma simultánea en una interfaz de red.
Para obtener instrucciones, consulte "Cómo configurar RIP" en la página 55 .

- Si está creando una interred IP grande, utilice OSPF.
Puede también ejecutar RIP y OSPF de forma simultánea; para obtener más información consulte "Configuración de OSPF" en la página 57 .


Para obtener información adicional sobre RIP y temas relacionados, consulte la NetWare 4.1 TCP/IP Reference .

Configuración de OSPF

OSPF se ha desarrollado para satisfacer la necesidad de un protocolo de encaminamiento escalable y de estándares abiertos para interredes IP de gran tamaño.
Como protocolo de estado del enlace, OSPF ofrece un encaminamiento muy eficaz y una convergencia rápida.

OSPF facilita la tarea de gestión de las interredes grandes ya que permite su división en dominios administrativos llamados áreas .
Estas áreas imponen una jerarquía en la red.
Todas las áreas OSPF se conectan a un área central principal mediante un Area Border Router o ABR ( router de borde de área).
El ABR comparte la información de encaminamiento OSPF entre el área y el segmento principal.

Al configurar un área OSPF, se le asigna un número decimal de cuatro bytes denominado ID de área.
También puede indicarse qué interfaces de red del router pertenecen a dicha área y si se trata de un área de pseudorutina.

NetWare 4.1 soporta el uso de enlaces virtuales entre routers OSPF.
Un enlace virtual "recompone" un segmento principal dividido en particiones, mediante la creación de un enlace directo punto a punto entre los ABR que conectan las áreas de segmento principal divididas a través del área de tránsito.

Ya que los enlaces virtuales son muy complejos y la probabilidad de configurarlos mal es muy alta, Novell desaconseja su utilización en las redes.

La mayoría de las interredes IP que se utilizan hoy en día no son redes OSPF "puras"; es decir, algunas de sus partes siguen utilizando otros protocolos de encaminamiento, como RIP.
OSPF utiliza un Autonomous System Boundary router o ASBR ( router de límite del sistema autónomo) para importar y propagar la información de encaminamiento de esos protocolos.
Al configurar OSPF, puede habilitar el router para que funcione como un ASBR.
Para que un ASBR importe rutas RIP aprendidas por medio de una interfaz, debe habilitarse el RIP en dicha interfaz.

Direccionamiento de AppleTalk

El direccionamiento de AppleTalk consta de un número de red y un número de nodo.
Cuando un nodo se conecta con la red, adquiere de forma dinámica un número exclusivo de 8 bits.
El nodo intenta utilizar la misma dirección cuando vuelve a conectarse con la red.
Si la dirección no está libre, intentará con otros números hasta que encuentre una dirección única.

La parte de la dirección correspondiente a la red, así como las asignaciones de zona, puede tomar dos formas, que dependen si la red es extendida o no extendida .

Una zona de AppleTalk es una agrupación lógica de dispositivos que permite a los usuarios localizar y acceder a servicios de la red tales como impresión.
Un nombre de zona es un nombre lógico asociado con algunos o con todos los servicios de una red o un conjunto de redes.

Una red no extendida soporta el direccionamiento de red de Fase 1.
En una red de fase 1, se asigna a cada red un solo número de red y una sola zona.
Esto limita el número total de nodos por red a 254.
Una red de Fase 1 no soporta el direccionamiento extendido de Fase 2 y sólo puede comunicarse con una red de Fase 2 si las redes están conectadas mediante un router.

Una red extendida admite el direccionamiento de red de Fase 2.
En una red de Fase 2 se asigna a cada red un rango de números de red, más que un solo número de red, y pueden asignarse hasta 255 nombres de zona.
Cada número de red soporta hasta 253 nodos.
Este esquema aumenta el número de nodos por red hasta el tamaño del rango en 253.

En las redes no extendidas , cada nodo adquiere el número de red asignado y busca en la red un número de red que no esté utilizado.
En las redes extendidas , cada nodo adquiere un número de red de inicio, que elige en un "rango de inicio" reservado (65.280 hasta 65.534) y después adquiere un número de nodo que no esté utilizado en la red.
Los números de inicio permiten que el nodo se comunique con el router y adquiera un número de red válido del rango configurado para la red.

Generación

La configuración de red y zona se propaga a través de routers generadores .
Un router generador puede enseñar a otros routers de una red la configuración de zonas y de red.
Un router no generador (o router de aprendizaje ) aprende la información de configuración del router generador.

En una red no extendida, el router generador se configura con el número de red y el nombre de zona de red.
En una red extendida, el router generador se configura con un rango de red y una lista de zonas de la red.

Al menos un router de la red debe ser un router generador.
El estado de generación se configura individualmente para cada interfaz.

Redes internas

Una red interna es una red virtual contenida dentro del módulo de AppleTalk; no tiene componentes físicos y para el servidor sencillamente es una de las redes con las que está conectado.
La red interna soporta dos nodos, el stack AppleTalk (nodo 1 de la red interna) y el router AppleTalk (nodo 2 de la red interna).

Los paquetes deben encaminarse desde una interfaz de la red externa hasta la red interna.
Puesto que la red interna requiere una dirección, toma un número de red.
Si se configura AppleTalk sin haber configurado una red interna, será necesario configurar una de las interfaces asociadas para permitir que se soporten las aplicaciones.

La red interna se configura de modo que esté habilitado el soporte de la aplicación AppleTalk.
La red interna es una red extendida y permite configurar múltiples zonas.
Si se elige configurar una única zona para la red interna, deberán notificarse todas las colas de impresión y demás servicios de un servidor que estén en una sola zona.
Con una red interna, los servicios pueden notificarse a cualquier zona de la red interna.
Esta red se configura siguiendo las mismas normas que para una red física externa, sujeta a las siguientes condiciones:
- Se requiere una rango de uno.

- No pueden especificarse múltiples zonas para la red interna de un router que utilice encaminamiento de transición.

- Los servicios de archivos AFP siempre se notifican en la zona por defecto.
Si se cuenta con muchos servidores NetWare AppleTalk, el número de redes internas podría dar lugar a un tráfico muy intenso de actualizaciones periódicas de encaminamiento.


A continuación se explican las ventajas de utilizar una red interna:
- La red interna tiene una distancia igual con cada interfaz asociada.
Esto asegura que en todo momento se utiliza la ruta óptima.

- Si el hardware de una interfaz asociada sufre una avería, las estaciones finales de la red aún podrán comunicarse con las aplicaciones si existe otra vía de acceso.


A continuación se explican las desventajas de utilizar una interfaz asociada para el soporte de aplicaciones en lugar de la red interna:
- Para cada router, no se necesita mantener una red adicional sólo para el soporte de aplicaciones .

- Los paquetes que se reciben en la interfaz seleccionada no requieren el salto adicional que se necesita para alcanzar la red interna.


Filtros

AppleTalk proporciona dos tipos de filtros: filtros para ocultar dispositivos y filtros de información de encaminamiento.

Los filtros para ocultar dispositivos restringen el acceso a los servicios filtrando los paquetes que notifican los servicios.
Los filtros de encaminamiento restringen el intercambio de información de encaminamiento entre los routers .
Los filtros de encaminamiento son de dos tipos: filtros de notificación de ruta y filtros de aceptación de ruta.

Los filtros de información de encaminamiento de AppleTalk son aplicables a zonas y redes, pero no pueden utilizarse para nodos específicos.

Habilite los filtros si desea que se restrinjan los paquetes que envía y recibe un router.
Si habilita esta opción, necesitará utilizar FILTCFG para configurar las opciones de filtrado.

- Si necesita AURP.
AURP permite encaminar los paquetes de AppleTalk a través de un túnel IP.
Además, a diferencia de RTMP, la información de encaminamiento solamente se intercambia cuando se produce un cambio en las topologías de las redes.
AURP es un método más económico para proporcionar la conectividad de AppleTalk entre dos emplazamientos, puesto que se utiliza menos ancho de banda para intercambiar la información de encaminamiento.
Para configurar AURP, utilice los parámetros Tunnel AppleTalk Through IP (AURP) y AURP.

- Si desea que el router sea un router generador.
Un router generador propaga la información de configuración de red y zona los demás routers de la red.
Un router no generador obtiene su información de configuración del router generador.
Al menos un router de una red AppleTalk debe ser un router generador.
El estado de generación se configura individualmente para cada interfaz.
No todos los routers generadores obtienen el número de red y el nombre de zona de los routers activos de la red.

Configuración de filtros 

La utilidad Filter Configuration o FILTCFG (Configuración de filtros) le permite configurar filtros que descartan de forma selectiva los paquetes que van a ser enviados o recibidos por un servidor o router.
Los filtros le permiten controlar la información sobre rutas y servicios que un servidor o router acepta o notifica.

Los filtros son útiles cuando se desea limitar algunos tipos de tráfico a ciertas partes de la topología de la red y para ofrecer un determinado nivel de seguridad.

- Filtros de información de servicios
- Filtros de información de encaminamiento
- Filtros de bridge

Tipos de filtros y conjuntos de protocolos

Filtros de reenvío de paquetes

Los filtros de reenvío de paquetes limitan el acceso a servicios concretos al evitar que el servidor o router reenvíe los paquetes de datos seleccionados.
Estos filtros ofrecen el más alto nivel de seguridad ya que examinan cada paquete de datos reenviado por el servidor o router.
Los paquetes se filtran basándose en lo siguiente:
- Origen del paquete
- Destino del paquete
- Tipo de paquete

Los filtros de reenvío de paquetes no restringen los paquetes de notificación de servicios enviados por los servidores.
Por lo tanto, los usuarios podrán ver notificaciones de los servicios aunque no puedan acceder a ellos.

Aunque los filtros de reenvío de paquetes ofrecen el más alto nivel de seguridad, podrían afectar al rendimiento del servidor o router ya que se aplican a cada paquete de datos que reciben.

Los filtros de reenvío de paquetes no se aplican a los paquetes que proceden de un router o están destinados a él.
Sólo se aplican a los paquetes reenviados, entre los que se incluyen los paquetes reenviados a la red IPX interna.

Se dispone de filtros de este tipo para los protocolos IPX y TCP/IP.

Filtros de información de servicios

Los filtros de información de servicios restringen la notificación o aceptación de servicios especificados filtrando la información de los paquetes de datos que notifican los servicios destinados a partes concretas de la red.
Estos filtros aumentan la seguridad ya que limitan la visibilidad de servicios seleccionados.
Sin embargo no ofrecen un nivel de seguridad tan alto como los de reenvío de paquetes ya que sólo examinan los paquetes de información de servicios.
También reducen el tráfico de red causado por los mensajes periódicos de información de servicios enviados por los servidores.

Se dispone de filtros de información de servicios para IPX (como filtros SAP entrantes y salientes) y AppleTalk (como filtros para ocultar dispositivos).

Filtros de información de encaminamiento

Los filtros de información de encaminamiento restringen el intercambio de información de encaminamiento entre routers al limitar las rutas añadidas a las tablas de rutas de los routers especificados.
Estos filtros aumentan la seguridad de la red ya que limitan la visibilidad de las redes especificadas.
Sin embargo, al igual que los filtros de información de servicios, estos filtros ofrecen un nivel de seguridad de red bajo ya que sólo examinan los paquetes de información de encaminamiento.
También reducen el tráfico de la red causado por el intercambio periódico de mensajes de información de encaminamiento entre los routers .

Se dispone de filtros de información de encaminamiento para IPX, AppleTalk y TCP/IP.

Hay dos tipos de filtros de información de encaminamiento: 
- Filtros de información de encaminamiento saliente que limitan las notificaciones de rutas enviadas por un router a un grupo de routers especificado.
- Filtros de información de encaminamiento entrante que limitan la aceptación de notificaciones de rutas recibidas por un router y procedentes de sus routers vecinos.

Filtros de información de encaminamiento saliente

Filtros de información de encaminamiento saliente que limitan la información notificada por un router a sus routers vecinos.
Cuando estos filtros están habilitados en un router, sólo se notifican las rutas permitidas.
Con esto, algunos routers y ciertas partes de la red no ven las rutas especificadas.

El filtro típico de este tipo consta de la ruta a la red de destino y la interfaz a través de la cual se envían las notificaciones de la ruta filtradas.
El filtro afecta a todos los routers de la red conectados a dicha interfaz.

Es posible que los filtros de información de encaminamiento saliente no surtan efecto si las estaciones finales que están en el mismo segmento LAN que el router de filtrado pueden acceder a las rutas filtradas.
Puesto que estos filtros sólo evitan la notificación de rutas filtradas a otros routers , pero no afectan a la tabla de rutas del router que filtra, éste entrega todos los paquetes que recibe con el destino especificado en las rutas filtradas.

Normalmente, las estaciones finales que están a un router de distancia, por lo menos, no pueden acceder a redes con rutas filtradas.

Para IPX, los filtros de información de encaminamiento afectan tanto a los clientes como a las rutas.
El cliente hará una petición de ruta que pase a través de los mismos filtros que la información de encaminamiento transmitida de forma rutinaria.
Sin embargo, si el cliente está conectado localmente a un router que está realizando filtrado de información de encaminamiento saliente y utiliza un software que puede usar encaminamiento estático, puede enviar paquetes a la red 1 a través del router R1 porque R1 tiene información sobre la red 1 en sus tablas de rutas.
Si sólo se llevan a cabo consultas de ruta IPX estándar (lo que sucede en la mayoría de las instalaciones), no puede existir una vía de acceso a las redes filtradas.

Filtros de información de encaminamiento entrante

Los filtros de información de encaminamiento entrante limitan la información de encaminamiento aceptada por un router y procedente de sus routers vecinos.

Cuando los filtros de este tipo están habilitados en un router, éste sólo acepta las rutas permitidas de cada router vecino, ocultando por ello las rutas especificadas procedentes de algunos routers y de ciertas partes de la red.
Sin embargo, estos filtros no pueden utilizarse para filtrar redes conectadas directamente.

El filtro de información de encaminamiento entrante más típico consta de la red de destino de la ruta y de la interfaz a través de la cual se espera recibir las notificaciones de la ruta.

Los filtros de este tipo evitan que el router de filtrado añada ciertas rutas a su tabla de rutas cuando recibe la información de sus vecinos.
El router de filtrado no puede reenviar un paquete a una ruta filtrada aunque reciba uno destinado a ella.
De esta forma, los filtros de información de encaminamiento entrante ofrecen un nivel de seguridad mayor que los de información de encaminamiento saliente.

Filtros para ocultar dispositivos de AppleTalk

Los filtros para ocultar dispositivos restringen la notificación de servicios en una interred mediante el filtrado de los paquetes que los notifican.
Estos filtros evitan que los usuarios localicen las direcciones de red de los servicios y ofrecen un nivel de seguridad de red.

En AppleTalk, el Protocolo de enlace de nombre (NBP) permite que los usuarios accedan a servicios del tipo servidores de archivos e impresoras.
De forma más concreta, permite al usuario o aplicación especificar parámetros de búsqueda como el nombre de la entidad de red y el tipo de servicio, así como la zona en la que deba llevarse a cabo la búsqueda.
La búsqueda se representa en una petición de consulta NBP enviada a la zona apropiada en la que podría estar el servicio.
Los servicios que coinciden con los parámetros de búsqueda responden directamente al usuario o aplicación que efectuaron la petición con la dirección AppleTalk del servicio.
Cuando el usuario o aplicación reciben la respuesta NBP, pueden comunicarse con el servicio por medio de la dirección de AppleTalk.

Cuando los filtros para ocultar dispositivos de AppleTalk están habilitados en un servidor o router, éste descarta las respuestas NBP para los servicios especificados (es decir, no entrega las respuestas al computador cliente o aplicación que las solicitó).
Por lo tanto, a esa parte de la red se le ocultan los servicios.

Un uso muy normal de NBP es la aplicación Selector de Macintosh* .
El usuario o aplicación emite una consulta NBP especificando una zona y tipo de servicio de interés.
La consulta se envía a la zona apropiada.
Todos los dispositivos o servicios del tipo especificado que haya en esa zona contestan con una respuesta NBP.
El Selector muestra la lista de dispositivos disponibles dependiendo de las respuestas NBP que reciba.
Por medio de la dirección de AppleTalk proporcionada en cada respuesta NBP, el usuario o aplicación puede comunicarse con el dispositivo o servicio.

Si está habilitado el filtro para la ubicación de ese dispositivo o servicio, el router descarta la respuesta NBP para que el usuario o aplicación no acceda a la dirección de red de dichos servicios.
Sin la respuesta NBP, la aplicación no puede conocer la existencia del dispositivo.

Los filtros para ocultar dispositivos ofrecen un nivel bajo de seguridad, pero tienen limitaciones.
Puesto que el filtro se habilita en un router, si el computador cliente que solicita la dirección está en la misma red que el servicio, la respuesta NBP va directamente al cliente y el router no tiene oportunidad de filtrarla.
Además, si la máquina cliente conoce la dirección de un servicio concreto, no necesita la respuesta NBP para acceder a él.

Puede configurar los routers AURP para que filtren la información de servicios que viaja a través de un túnel IP.
Si se habilita un filtro en el túnel, todas las redes a las que se puede acceder por medio del túnel se ven afectadas por los filtros de información de servicios configurados para el router AURP.
Consulte el Capítulo 4, "Configuración de AppleTalk," para obtener más información sobre la configuración de AURP.

Uso de la consola IPX

La consola IPX (IPXCON) es una utilidad de consola de diagnósticos que proporciona acceso a estadísticas e información acerca del estado de diversos componentes del protocolo IPX.
IPXCON utiliza SNMP para acceder a esta información desde cualquier sistema local o remoto de la red.
IPXCON trabaja en redes IPX y TCP/IP y utiliza UDP para ejecutarse en redes IPX o TCP/IP.

En esta sección se describe la interfaz para IPXCON.
Puede utilizar IPXCON para realizar las siguientes tareas:
- Monitorizar y localizar problemas en segmentos de red y routers IPX de toda la interred IPX.
- Visualizar el estado de cualquier router IPX o segmento de red de la interred.
- Visualizar todas las vías de acceso a través de las cuales pueden circular los paquetes IPX.
- Localizar todos los routers IPX activos de la interred
- Visualizar los circuitos operativos para IPX 
- Monitorizar los routers remotos en los que se ejecuta el software NetWare IPX router tm. 

En general, el nodo NetWare IPX local es el objetivo de IPXCON.
No obstante, puede accederse a cualquier otro nodo de router IPX que soporte SNMP si se cambia la dirección del host.
Esto incluye las stacks de protocolos IPX provistas por otros proveedores.

Configuración de protocolos de comunicación en un servidor SFT III

En este capítulo se describe cómo se utiliza INETCFG para configurar los protocolos de comunicación AppleTalk, IP e IPX tm para un servidor SFT III de NetWare® .
Para ejecutar esta configuración, la utilidad INETCFG debe ejecutarse en un motor MS y en motores de E/S de los servidores tanto primarios como secundarios.

El motor de E/S del servidor primario utiliza los protocolos de comunicación configurados con INETCFG para ejecutar todas las funciones de encaminamiento de red de su servidor SFT III.
Asimismo, encamina los paquetes por una red virtual separada que conecta los motores de E/S al motor MS.
Esta red virtual se denomina "Red SFT III".
Los protocolos de comunicación del motor de E/S del servidor secundario se configuran de forma que puedan ejecutar funciones de encaminamiento de red dirigidas a la red SFT III si el motor de E/S del servidor primario falla.

Aunque el software del protocolo de comunicación instalado en el servidor SFT III es el mismo que el software del protocolo estándar, la configuración puede variar en algunos aspectos.
Esto se debe a que es necesario configurar dicho software en cada uno de los motores del servidor SFT III.

Creación de una red SFT III para AppleTalk

Cuando se configura AppleTalk en un motor MS y en motores de E/S de un servidor SFT III, es necesario que exista una red AppleTalk virtual de SFT III.
Esto significa que AppleTalk debe estar conectado a esa red AppleTalk virtual de SFT III en cada uno de estos tres motores de servidor SFT III y que debe asignarse un número de red AppleTalk separado a la red virtual, como se muestra en la Figura 8-2 .
En los motores de E/S, AppleTalk debe configurarse como routers , con el reenvío de paquetes habilitado (On) .
En el motor, MS AppleTalk debe configurarse como sistema final, con el reenvío de paquetes inhabilitado (Off).

Figura

Configuración de red AppleTalk interna de SFT III

Asimismo, la red AppleTalk que conecta los motores de E/S al motor MS debe estar asignado a una o varias zonas AppleTalk.
La lista de zonas puede contener las zonas existentes o zonas exclusivas que contengan sólo la red SFT III.

Una de las consecuencias de esta configuración es que se genera una asociación de la tarjeta virtual de SFT III en cada uno de los motores, así como una línea en cada uno de los archivos NETINFO.CFG para indicar esta asociación.
En el motor MS, se crea el comando siguiente:

BIND AppleTlk MSENGINE 

En el motor de E/S, el comando BIND incluye el número de la red AppleTalk.
Por ejemplo, si configura un motor de E/S de un servidor SFT III con el número de red AppleTalk 1234, el comando BIND será como el que se indica a continuación:

BIND AppleTlk MSENGINE net= 1234-1234 .

Creación de una red SFT III para IPX

Con el protocolo IPX no es necesario crear una red interna dado que ya existe.
Cada servidor NetWare dispone de su propia red IPX interna.
En un servidor SFT III, ambos motores MS comparten una misma red interna y cada uno de los motores de E/S tiene su propia red IPX interna.
El motor de E/S del servidor SFT III primario se asocia automáticamente a la red IPX interna del motor MS.
Por este motivo no es necesaria una red interna adicional para conectar los dos motores del servidor SFT III.

Creación de una red SFT III para IP

Cuando se configura el encaminamiento de IP en un motor MS y en motores de E/S de un servidor SFT III, es necesario que exista una red SFT III .
Esta red virtual conecta el motor MS con los motores de E/S.
Esto significa que debe crearse una subred separada con dos direcciones de host separadas, como se muestra en la Figura 8-3.

Figura

Configuración de red IP interna de SFT III

Dado que los motores MS están duplicados, pueden compartir la misma dirección de host.
De modo parecido, conviene configurar cada motor de E/S con la misma dirección de host.
Esto puede realizarse sin que se produzca ningún conflicto, ya que sólo uno de los motores de E/S funciona cada vez en la red virtual SFT III.

Esta subred se comporta en la red IP como cualquier otra subred y debe tener asignada una dirección IP perteneciente a las direcciones de red IP que el usuario tenga asignadas.

Una de las consecuencias de esta configuración es que se genera una asociación con una tarjeta virtual en cada uno de los motores, así como una línea en cada uno de los archivos NETINFO.CFG para indicar esta asociación.
Por ejemplo, si configura una dirección del host de la red de SFT III de un motor del servidor con el número 144.1.1.2 y su máscara con el valor FF.FF.0.0, el comando BIND será como el que se indica a continuación:

BIND IP MSENGINE Address=144.1.1.2 Mask=FF.FF.0.0 

Configuración de los protocolos de comunicación en los motores del servidor SFT III

La configuración de los protocolos de comunicación en los servidores SFT III implica la ejecución de INETCFG para establecer los parámetros de los protocolos individuales.
Si la asignación de la memoria entre el motor MS y el motor de E/S no tiene establecido el ajuste adecuado, INETCFG no se ejecutará.

Si los servidores SFT III tienen 16 MB de memoria o menos, la memoria asignada no será suficiente para que el motor de E/S ejecute INETCFG.

Para establecer la asignación de memoria entre el motor MS y el de E/S al nivel adecuado para que se ejecute INETCFG, introduzca los comando siguientes en el archivo IOSTART.NCF : 

SET NEW START ADDRESS FOR UNCLAIMED MEMORY BLOCK=8000000 SET MAXIMUM PHYSICAL RECEIVE PACKET SIZE=1518 

En el motor de E/S, 1518 es lo adecuado para Ethernet; 4202 es lo conveniente para token ring .

Apague el servidor y vuelva a arrancarlo para que se activen los cambios introducidos en la asignación de memoria.

Cuando se haya establecido la asignación de memoria adecuada entre el motor MS y el motor de E/S, INETCFG podrá ejecutarse según lo descrito.

Los parámetros del protocolo de comunicación se configurarán entonces por separado en cada uno de los motores siguientes del servidor SFT III:
- Motor MS
- Motor de E/S del servidor primario

Tarjetas de interfaz

Los aspectos siguientes están relacionados con problemas de las tarjetas de interfaz:
- Utilice tarjetas certificadas por el grupo Novell® Labs tm. 
- Si está utilizando tarjetas token ring IBM*, se recomienda una RAM compartida de 16 KB.
- En la mayoría de los casos, las tarjetas LAN pueden instalarse con los ajustes de fábrica por defecto para Video Memory Addressing.
Sin embargo, los ajustes por defecto de algunas tarjetas pueden entrar en conflicto con tarjetas VGA/EGA ya instaladas.
Para resolver el conflicto, deberá volver a configurarse la tarjeta VGA/EGA, o bien la tarjeta de interfaz de red.

- El software IPX no puede encaminarse en dos tarjetas de interfaz distintas conectadas a la misma red IPX de rutas.
En el caso de redes token ring , en vez del encaminamiento puede utilizarse el paso por un bridge de la ruta de origen para conectar dos o más redes IPX.

- Si está ejecutando los encaminamientos en una tarjeta Ethernet, ajuste el tamaño máximo de paquetes recibidos físicamente en al menos 1524 en el archivo STARTUP.NCF .
Si está ejecutando los encaminamientos en una tarjeta token ring , ajuste el valor en al menos 4510.

- Antes de instalar una tarjeta LAN, asegúrese de que comprueba todas las interrupciones disponibles, las direcciones base de E/S y la dirección de asignación de memoria de todas las tarjetas.
Para evitar que se creen conflictos con los ajustes de los parámetros de la tarjeta, compruebe que no haya varias tarjetas que utilicen los mismos valores.
Como referencia, los valores típicos utilizados en un PC son los que se indican a continuación:
- COM1 utiliza IRQ-4 , puerto=3F8-3FF 
- COM2 utiliza IRQ-3 , puerto=2F8-2FF 
- LPT1 utiliza IRQ-7 , puerto=378-37F 
- LPT2 utiliza IRQ-5 , puerto=278- 27F 
Consulte el manual del hardware de cada dispositivo para determinar los ajustes válidos de IRQ, la base de E/S y las ubicaciones de memoria.
A medida que configure los enlaces, registre esta información en una hoja de trabajo para evitar que se produzcan conflictos.

- Cuando se emite un comando para descargar un controlador de tarjeta, se descargan todas las peticiones de esa tarjeta.
Por ejemplo, si se han cargado dos tarjetas Synchronous/+, el comando UNLOAD SYNCLPUS descargará ambas tarjetas y todas las llamadas a estas tarjetas se darán por terminadas.

Configuración

Los elementos siguientes están relacionados con problemas de configuración del producto:
- Obtenga los archivos .LAN y .LDI más recientes de NetWare de su distribuidor de tarjetas de interfaz de red.
- Utilice siempre INETCFG para configurar el producto de interred NetWare 4.1.
- INETCFG proporciona para la configuración un conjunto de menús fáciles de utilizar.
INETCFG cambia automáticamente los comandos LOAD y BIND utilizados con NetWare 3.12, NetWare 4.1 y versiones anteriores.
Asimismo, INETCFG permite ver y, en algunos casos, editar (para la resolución de problemas) el archivo de información sobre la configuración ( NETINFO.CFG ), el archivo AUTOEXEC.NCF y los mensajes generados durante el reinicio del sistema.
Finalmente, INETCFG proporciona varios niveles de ayuda sensible al contexto para facilitar y acelerar el proceso de configuración.

- No modifique el archivo NETINFO.CFG ubicado en el subdirectorio SYS:ETC .
- El fichero NETINFO.CFG contiene los comandos LOAD y BIND que ha configurado con INETCFG.
Si emplea la utilidad INSTALL para crear un archivo AUTOEXEC.NCF después de haber utilizado INETCFG, los comandos LOAD y BIND de la memoria del sistema se grabarán en el archivo AUTOEXEC.NCF .
Novell recomienda que no se utilice este procedimiento ya que puede crear entradas duplicadas de los comandos LOAD y BIND ya existentes en el archivo NETINFO.CFG .

- Si el archivo NETINFO.CFG aparece con el indicador READ ONLY y el usuario cambia un campo en INETCFG (por ejemplo, IPX Network Number en Bindings ), este cambio se reflejará en INETCFG, pero no en NETINFO.CFG .
El archivo NETINFO.CFG no sufrirá ningún cambio ya que es READ ONLY .
Sin embargo, INETCFG avisa cuando esto se produce, señalando que el archivo NETINFO.CFG no ha podido cambiarse.
Para guardar el cambio, asigne a INETINFO.CFG el indicador READ/WRITE , vuelva a INETCFG e introduzca de nuevo el cambio.

- Asegúrese de configurar los tipos de trama correctos cuando asocie IP, IPX y AppleTalk a una interfaz de LAN.
Tenga en cuenta que el tipo de trama por defecto para IPX se ha cambiado a Ethernet_802.2.

- No cargue ni asocie el bridge desde la línea de comandos.

Consejos para la resolución de los problemas

En esta sección se presentan las recomendaciones sobre la configuración de AppleTalk:
- Incremente el tiempo de Routing Update Timer en INETCFG de los 10 segundos por defecto a cualquier otra cifra entre 60 y 300 segundos cuando utilice enlaces con menos de 19.2 Kbps (vía de acceso del parámetro: Seleccione Bindings > Seleccione una asociación específica ).
De esta forma se aumenta al máximo el rendimientos de datos ya que se reduce al mínimo la información de encaminamiento enviada a través del enlace.
Al aumentar al máximo el tiempo de Router Update Timer aumentará también el tiempo que la red tarda en convergir tras un cambio de topología.

- Si una red interna está configurada en un NetWare para un servidor AppleTalk, el nombre NBP (Name Binding Protocol - Protocolo de enlace de nombre) del servidor reside en la zona por defecto de la red interna.
Si la red interna no está configurada, el NetWare para el nombre del servidor AppleTalk reside en la zona por defecto de la interfaz LAN seleccionada asociada a AppleTalk.

- Cuando se configura el estado de generación de AppleTalk in INETCFG (vía de acceso del parámetro: Seleccione Bindings > Seleccione una asociación específica > Seleccione Network Range and Zone configuration ), debe comprobar que exista al menos un router generador en cada red AppleTalk.
Si existen otros routers generadores conectados a la misma red, todos los números de red y nombres de zona deberán coincidir.


Puntos a comprobar para resolver los problemas

Para localizar y resolver los problemas de AppleTalk, lleve a cabo los pasos siguientes:

Utilice la prueba Echo para determinar si el router puede comunicarse con un nodo concreto de una de las redes.

Cuando ejecute la prueba de Echo, el router enviará el número de paquetes especificado al nodo especificado a través de la red, tras lo cual el nodo devolverá el mismo número de paquetes.

Si el campo Number of Packets Received no incrementa, puede que el nodo especificado esté apagado o que el cable de conexión se haya soltado.

Si el resultado de la prueba de Echo es satisfactorio, el número de paquetes recibidos será igual al número de enviados.
Si la comunicación es satisfactoria, AppleTalk está funcionando con esa interfaz y el nodo especificado está habilitado y funcionando.
Para ejecutar una prueba de Echo:

Seleccione Lookup Network Entities en el menú principal de ATCON.

Introduzca un nombre de zona en la opción Name.

Seleccione la opción Proceed .

Seleccione una entidad en la lista visualizada.

Edite los siguientes parámetros de prueba según proceda: Mode, Number of Packets to Send , Timeout y Number of Data Bytes . 

Utilice una prueba de Echo para determinar si el router puede comunicarse con un nodo en una red AppleTalk remota.

Utilice el mismo procedimiento que se describe en el Paso 1 .

Si la prueba es satisfactoria, el router puede acceder a otra red AppleTalk y todos los routers AppleTalk que se encuentran entre ambos están realizando los encaminamientos correctamente.

Si la prueba no es satisfactoria, ejecute una prueba de eco con el router más próximo en la vía de acceso al nodo remoto.
Siga ejecutando las pruebas de eco con cada router de la vía de acceso hasta que localice el punto donde la prueba falla.
Tome las medidas adecuadas con el router donde la prueba de eco falla.

Compruebe la tabla de rutas en ATCON.

La opción Routing Table contiene entradas de todas las redes y zonas a las que el router puede enviar los paquetes.

Problemas comunes

En esta sección se presentan los varios síntomas de los problemas comunes y sus soluciones potenciales.
Los problemas los divide en categorías el archivo NLM que los produce.

MACIPXGW.NLM
- MACIPXGW no se carga y aparece el mensaje de error Failed to find zone for gateway .
- Intente cargar MACIPXGW manualmente desde la consola.
Si se carga es que MACIPXGW estaba cargado antes de que la Stack AppleTalk pudiera resolver la zona por defecto.
Retrase todo lo posible la carga de MACIPXGW en el archivo AUTOEXEC.NCF .
Intente primero cargar AppleTalk, a continuación los demás protocolos y finalmente MACIPXGW.

- Si MACIPXGW no se carga aún habiéndolo intentado manualmente, puede que la stack AppleTalk se encuentre en una interfaz externa no generadora y que no exista un router en la red a la que la interfaz esté conectada.
Vuelva a configurar AppleTalk para que sea un router generador en esa interfaz, o bien habilite la red interna AppleTalk.

- MACIPXGW no se carga porque la versión API de AppleTalk no es la correcta.
En el computador tiene archivos APPLETLK.NLM y MACIPXGW.LAN incompatibles.
Actualice ambos archivos con la última versión.

- Aparece el mensaje Gateway name truncated to 32 characters .
El nombre del gateway que ha configurado tiene más de 32 bytes.
Este error no es fatal, pero deberá cambiarlo para que tenga menos de 32 bytes con el fin de evitar que vuelva a aparecer el mensaje de advertencia.

- Existen demasiados paquetes de difusión IPX tm. encapsulados en paquetes AppleTalk de unidifusión de la red y desea reducir el tráfico. Disminuya el número del parámetro Unicast Threshold de MACIPXGW para forzar que se encapsulen los paquetes de difusión IPX en paquetes de difusión AppleTalk.


Zonas y nombres de zona
- Las zonas reconfiguradas siguen siendo visibles en el Chooser del cliente o en las tablas de zona del router.
Este problema tiene su origen en cómo AppleTalk mantiene las información sobre las zonas y las tablas de rutas en la interred.
Los routers no reciben automáticamente la información sobre los cambios en la configuración de las zonas.

- Si este problema se produce en su propia posición y el router reconfigurado es el único que está directamente conectado a la red que soporta las nuevas zonas, apague el router y déjelo apagado de la red al menos 20 minutos (o hasta que las rutas cambiadas interrumpan su servicio en la interred).
Los zonas adquirirán validez cuando vuelva a encenderse el router.

- Si más de un router está conectado a la red que soporta nuevas zonas, todos deberán estar configurados para que soporten las mismas zonas.
Apague todos los router, reconfigure las zonas y déjelos apagados al menos 20 minutos (o hasta que las rutas cambiadas interrumpan su servicio en la interred).
Las nuevas zonas adquirirán validez cuando vuelvan a encenderse los routers.

- Si las zonas están soportando más de una red y desea cambiar las zonas de todas o algunas de estas redes, apague los routers conectados a cada una de las redes de las que quiere cambiar las zonas.
Reconfigure las zonas y déjelos apagados durante al menos 20 minutos (o hasta que las rutas cambiadas interrumpan su servicio en la interred).

- Los nombres de zona no coinciden con los nombres de zona configurados.
- Si en Chooser o ATCON aparecen nombres de zona adicionales después de que se haya encendido el router en una red AppleTalk, compruebe si ha escrito los mismos nombres de zona en los distintos archivos de configuración utilizando las mismas mayúsculas y minúsculas, ya que la distinción entre estos dos tipos de letra hace que una misma palabra tenga distintos nombres de zona.
- Verifique que no existan espacios en blanco finales en los nombres de zona.
- Un número de red o un rango de red tiene una lista de zona distinta en routers distintos después de que la lista de zona de la red se cambiara en el router generador.
El router generador se ha encendido antes de que la red interrumpiera el servicio en todos los routers.
Apague el router generador y los routers directamente conectados, y espere a que la red interrumpa el servicio de todos los routers 
El tiempo recomendado está entre 10 y 20 minutos, dependiendo del tamaño de la interred AppleTalk.
Deje un minuto por salto.
Si el router más lejano está a 11 saltos de distancia del router generador donde ha cambiado la red, espere al menos 11 minutos antes de volver a encender el router generador y los router directamente conectados.


Mensajes de consola

La consola presenta mensajes avisando sobre la falta de coincidencia entre los números de red o las zonas.

Soluciones generales
- Verifique que el número o rango de red sea exclusivo en la interred.

- Verifique que todos los routers del cable coincidan en el número o rango de red.

- Verifique que la lista de zonas de la red tenga los mismos nombres y que todos los routers coincidan en ese nombre.


Soluciones para redes extendidas
- Verifique que el rango de la red extendida del router en modo de transición tenga un rango de red correspondiente a uno (por ejemplo 21-21).

- Verifique que el rango de la red extendida es exclusivo en la interred.

- Verifique que todos los routers del cable coincidan en el rango de red.

- Verifique que todos los routers del cable coincidan en los nombres de zona.


Problemas de asociación
- AppleTalk no consigue asociarse a una interfaz generadora.
Dado que AppleTalk de NetWare 4.1® efectúa una generación lógica, AppleTalk verifica la información sobre la generación configurada con la configuración de la red actual perteneciente a otros routers de esa red.
Si la información sobre la generación no es consistente con la configuración actual de la red, significa que existe un fallo en la configuración y que AppleTalk no permite que la interfaz funcione.
Compruebe la configuración de la interfaz para verificar que el número/rango de la red y los nombres de zona estén configurados exactamente igual que en todos los demás routers generadores conectados a la misma red.

- AppleTalk no consigue enlazarse a una interfaz y aparece el mensaje de error Error binding APPLETALK to boardboard_name .
Esto se produce cuando el controlador no se había inicializado completamente cuando AppleTalk fue a asociarse la primera vez.
Esto puede pasar cuando el comando LOAD driver se encuentra justo encima del comando para asociar AppleTalk con el controlador en un archivo de procesamiento por lotes como AUTOEXEC.NCF .
Para resolver este problema, utilice INETCFG para configurar la tarjeta y AppleTalk, o bien sitúe manualmente la línea LOAD driver encima de la línea LOAD APPLETLK para que la tarjeta tenga más tiempo para completar la inicialización.


Problemas de impresión y de servicio
- Los clientes de AppleTalk no pueden ver los servicios facilitados en el servidor NetWare desde la opción Chooser y el servidor no contesta a las peticiones ARP.
El controlador LAN no soporta la multidifusión correctamente.
Utilice un controlador LAN distinto.

- Los clientes de AppleTalk no pueden ver en Chooser los servicios proporcionados en el servidor NetWare.
Cargue ATCON en la consola NetWare y, a continuación, seleccione Protocol Configuration .
Si el estado de la stack es Disabled y el router AppleTalk no tiene configurada una red interna, puede que haya olvidado seleccionar una interfaz LAN para proporcionar el soporte a las aplicaciones cuando se asocia AppleTalk a la interfaz.

- Existen diferencias en las tablas de rutas entre los routers.
Las tablas de rutas entre los routers AppleTalk pueden ser diferentes.
Compruebe Static Routes for On-demand Calls en INETCFG (vía de acceso del perímetro: Seleccione Protocols), los filtros de encaminamiento entre los routers y la lista de pares AURP.


Resolución de problemas de IP

Esta sección contiene información para la resolución de problemas de IP dividida en tres categorías:
- Herramientas para la resolución de problemas 
- Puntos a comprobar para resolver los problemas 
- Problemas comunes 

Si el problema es de tipo general, ejecute el procedimiento descrito en " Puntos a comprobar para resolver los problemas " para localizar y resolver el problema.
Cuando se presenta un problema con un síntoma específico, consulte la información facilitada en " Problemas comunes ".

Herramientas para la resolución de problemas

TCPCON es una utilidad NLM que proporciona el acceso a las estadísticas y la información sobre el estado de los distintos componentes si el protocolo TCP/IP es adecuado.
TCPCON utiliza SNMP para acceder a esta información desde cualquier sistema local o remoto de la red.
TCPCON funciona en redes TCP/IP e IPX.
Introduzca LOAD TCPCON en el indicador de la consola del sistema.
Para monitorizar un sistema remoto, seleccione la opción Change Host e introduzca la dirección IP del host remoto que quiere probar.
Si se presenta la información detallada sobre el host remoto, quiere decir que dicho host está habilitado y funcionando.
Puede utilizar IPXCON para ejecutar las tareas siguientes:
- Monitorizar la actividad en los segmentos de red TCP/IP de la interred
- Mostrar la información sobre la configuración y las estadísticas sobre los protocolos TCP/IP siguientes: IP, ICMP, UDP, TCP, OSPF y EGP
- Mostrar las rutas IP actualmente conocidas en un nodo TCP/IP 
- Mostrar las interfaces de red que un nodo TCP/IP soporta 
- Acceder al registro de alerta que SNMPLOG mantiene (únicamente para sistemas locales)
- Acceder a la información de TCP/IP en cualquier stack de protocolo remoto que soporte la MIB (Management Information Base-Base de información de gestión) de TCP/IP.
TCPCON requiere que se cargue SNMP en el host remoto; de lo contrario, aparecerá un mensaje de error avisando que el host no está disponible.
Otro motivo por el que el mensaje Host unavailable puede aparecer es que se haya producido un error de encaminamiento.
Puede comprobar la tabla de rutas y aceptar el valor 127.0.0.1 en la opción Change Host .
Seleccione Routing Table para ver la tabla con la información sobre las rutas que el software NetWare 4.1 ha recibido de los protocolos de encaminamiento (RIP y OSPF) o de las rutas estáticas.
Compárelo con la tipología de la dirección de la red.
Para más información, consulte la NetWare 4.1 TCP/IP Reference .


Puntos a comprobar para resolver los problemas

Para localizar y resolver los problemas de TCP/IP, lleve a cabo los pasos siguientes:
- Utilice TCPCON para verificar que los paquetes IP se están enviando y recibiendo.

- Verifique que IP Status esté en Packet Forwarding.

- Utilice PING o TPING para probar la conectividad.

- Verifique que el encaminamiento estático esté configurado si a la red se han conectado routers de terceras partes que no utilizan RIP ni OSPF.
