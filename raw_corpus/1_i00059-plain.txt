
Guía para el usuario del cliente DOS y MS Windows

Cómo utilizar este manual

Introducción

La Guía para el usuario del cliente DOS y MS Windows NetWare® tiene tres finalidades: ayudarle a configurar y conectar sus estaciones de trabajo cliente a la red e introducir las herramientas del usuario cliente para gestionar el acceso y las conexiones a una red NetWare.

Este manual está dirigido a los usuarios y supervisores no familiarizados con la conectividad de cliente NetWare y los supervisores de la red responsables de configurar y gestionar las estaciones de trabajo cliente NetWare.

Este manual incluye conceptos y procedimientos para la instalación y el uso de software de cliente NetWare en redes NetWare 2, 3 y 4.
Existen referencias para cada versión de NetWare.
Ignore cualquier referencia que no pertenezca a la versión de NetWare a la que se encuentra conectado.

Utilice la Guía de referencia técnica del cliente DOS y MS Windows NetWare para las configuraciones avanzadas y la resolución de problemas de sus estaciones de trabajo Cliente NetWare.

Comandos

Los carácteres en negrita indican los elementos que deben escribirse, tales como comandos y opciones.

En ejemplos de sintaxis, una barra delimitadora que separa dos opciones de comandos indica que se puede seleccionar cualquiera de las dos.

Por ejemplo: -S &verbar; -R

Los comandos DOS y las letras de opciones de comandos aparecen en mayúsculas.
Por ejemplo: FTPD .

Los nombres de archivo, nombres de directorio y vías de acceso DOS se muestran en mayúsculas.
Por ejemplo, AUTOEXEC.@BAT. 

Debido a que DOS no tiene distinción de mayúsculas y minúsculas, puede escribir comandos en cualquiera de las dos formas.

Los puntos suspensivos en los ejemplos de sintaxis indican que se pueden repetir los parámetros, opciones o las configuraciones.

Conceptos básicos

Descripción general

El software Cliente TM NetWare® para DOS y Microsoft* (MS) Windows proporciona conectividad básica y acceso a los recursos de red para todas las estaciones de trabajo de DOS y MS Windows.
Además, proporciona servicios avanzados de red, como por ejemplo, transferencia rápida de datos, gestión de impresión y archivos y acceso a los Servicios del Directorio NetWare TM para compartir y producir información de manera efectiva.

El software del Cliente NetWare se ha rediseñado para dar un mejor soporte a las cada vez más importantes necesidades de un mayor rendimiento y funcionalidad en las estaciones de trabajo para entornos de DOS y MS Windows.
Este nuevo diseño introduce un componente de software para el entorno de DOS conocido como el programa Requester TM DOS de NetWare.
El Requester DOS de NetWare sustituye los archivos del shell de NetWare, como por ejemplo NETX.COM.

El programa de herramientas del usuario NetWare también se ha rediseñado para permitir un mejor acceso y gestión de los servicios de NetWare para las estaciones de trabajo cliente para DOS y MS Windows.

Introducción

Para aprovechar las estaciones clientes de NetWare ®, deberá entender cómo se relacionan con el sistema operativo NetWare al que están conectadas y revisar los conceptos básicos de la conectividad.

Los usuarios conectados a NetWare 2 y NetWare 3 TM deben comprender cómo sus estaciones clientes se relacionan con las redes de servidores Bindery de NetWare.

Los usuarios conectados a la red de NetWare 4 TM deben comprender la arquitectura de los Servicios del Directorio NetWare TM (NDS) y cómo ésta afecta a su entorno.

Tipos de red

Una red enlaza dos o más computadores para que compartan recursos y periféricos (tales como unidades de disco, impresoras y datos).
Hay dos categorías de red: par a par y de cliente-servidor. 

Redes par a par

La red par a par más sencilla tiene dos estaciones de trabajo conectadas entre sí que pueden usar los recursos de ambas. 
Todas las estaciones de trabajo de una red par a par son (o pueden ser) iguales.
No se designa ninguna de ellas como el computador principal o el centro de control.

Redes cliente-servidor

La red cliente-servidor tiene al menos una estación de trabajo configurada como un servidor de NetWare.
Como se muestra en la Figura 1-1 en la página 5, el servidor es un computador que ejecuta el sistema operativo de la red.
El servidor controla la comunicación y los recursos de red compartidos.
Las estaciones clientes son los computadores individuales conectados a la red.

Conexiones de la red

Deberá realizar dos tipos de conexiones para usar los recursos compartidos de la red:

- Una conexión física
- Una conexión lógica

Conexión física

Para que los recursos y la información de una red sean compartidos, los servidores, las estaciones clientes, y los demás dispositivos de la red deben estar conectados físicamente.

Todos los dispositivos deben tener una tarjeta de red.
Un cable conectado a la tarjeta de red proporciona la vía de acceso a la comunicación, como se muestra en la Figura 1-2 en la página 6 .

Figura

Tarjeta de red y cableado

La conexión física determina el tipo de cable usado, la configuración del cable (llamada la topología de la red) y la forma en que la información se transmite por el cable (denominada protocolo ).

Conexión lógica

Una vez conectado físicamente a la red, deberá realizar una conexión lógica para acceder a los servicios de ésta.

Para realizar la conexión lógica en una red de NetWare, deberá demostrar que es un usuario autorizado.
Para ello al entrar en la la red deberá proporcionar al sistema operativo de NetWare la identificación de usuario y una contraseña válida.
Al entrar en una red o conectarse a un servidor, ha hecho una conexión lógica.

Recursos de la red

Una vez conectado a una red de NetWare, tendrá acceso a los recursos físicos y lógicos a través de ésta.
Se guarda una definición para cada recurso en una base de datos usada por el sistema operativo de NetWare.

Un ejemplo de ello es una cola de impresión o un nombre de grupo.
Cómo interactúan con la base de datos de recursos depende en gran manera del sistema operativo NetWare que esté ejecutando. 

Recursos de NetWare 2 y NetWare 3

Los sistemas operativos de NetWare 2 y NetWare 3 utilizan una estructura de base de datos simple llamada un Bindery para gestionar las definiciones de los recursos de la red.
En el Bindery guarda una definición para cada recurso de la red.

Por ejemplo, hay una definición para la información de usuario, como el nombre del usuario y la contraseña.
Hay un Bindery para cada servidor de la red que es independiente de las definiciones de recursos creadas en otros servidores.

La estructura del Bindery para cada servidor es llana, con una ubicación única para acceder y gestionar los recursos de la red.
Por consiguiente, cuando usa los recursos en una red con varios servidores, necesita conocer sus ubicaciones físicas y lógicas.

Recursos de NetWare 4

NetWare 4 utiliza una estructura de base de datos global llamada Servicios del Directorio NetWare ( NDS TM ) para gestionar las definiciones de los recursos.
Los NDS tratan a todos los recursos de la red como objetos de una base de datos distribuida conocida como la base de datos del Directorio .
Esta organiza los recursos en una estructura de árbol jerárquica, independiente de su ubicación física.

La base de datos del Directorio soporta una red de servidores entera.
En vez de almacenar toda la información en cada servidor individual como en NetWare 2 y 3, la información se distribuye en una base de datos global a la que pueden acceder todos los servidores de NetWare.

Los usuarios pueden acceder a cualquier servicio de la red, como los servicios de archivos o de impresión, sin necesidad de conocer la ubicación física del servidor que proporciona el servicio.

Componentes de un árbol del Directorio

Los Servicios del Directorio NetWare operan en una organización lógica llamada el árbol del Directorio . 
Se le llama así porque los objetos están almacenados en una estructura de árbol jerárquica que empieza con el objeto &lsqb;Root&rsqb; y se ramifica. 

El árbol del Directorio lo forman tres tipos de objetos: 

- El objeto &lsqb;Root&rsqb;. El programa de instalación coloca el objeto &lsqb;Root&rsqb; en la parte superior del árbol del Directorio.

- Objetos contenedores.
Una rama del árbol del Directorio consiste en un objeto contenedor y todos los objetos que retiene, que pueden incluir otros objetos contenedores.

- Objetos Hoja.
Los objetos Hoja se encuentran al final de las ramas y no contienen ningún otro objeto.
Volumen, Usuario, y Objetos Servidores NetWare son algunos ejemplos de Objetos Hoja.

De esta forma se ofrece a los usuarios una visión más manejable del árbol y no necesitan saber dónde está ubicado cada objeto.

La estructura del sistema de archivo

El Bindery y los Servicios del Directorio NetWare le ayudan a gestionar los recursos de la red tales como los servidores de NetWare y las impresoras, pero no proporcionan un control sobre el sistema de archivo (volúmenes, directorios y archivos).
Las utilidades de texto y gráficos le ayudan a gestionar tanto los recursos de la red como el sistema de archivo.

La estructura del sistema de archivo está separada de la estructura del Bindery o de los Servicios del Directorio NetWare.
Físicamente, toda la información de la red se almacena en discos duros controlados por un servidor; pero lógicamente, el sistema de archivo se representa en NetWare 2 y 3 como un volumen y en NetWare 4 como un objeto Volumen. 
Este volumen lo gestiona la emulación del Bindery de NetWare 4.

El sistema de archivo lo conforman tres tipos de estructuras: los volúmenes, los directorios, y los archivos.

La estructura de archivos de NetWare 2 y NetWare 3

La estructura de archivo de NetWare 2 y NetWare 3 empieza con el servidor de archivos.
Los discos duros físicos en un servidor de archivos están divididos en volúmenes, como si fueran una colección de libros.
Un volumen puede contener varios discos duros físicos, o un disco duro puede dividirse en varios volúmenes. 

Un volumen marca la base de cada estructura de directorio.
Parece y actúa como un directorio raíz o una unidad de disco de DOS.
Los directorios están almacenados en los niveles del volumen y de los subdirectorios.
Los archivos están almacenados en directorios y subdirectorios de la misma manera en que los sistemas operativos de DOS almacenan los archivos.

La estructura de archivo de NetWare 4

La estructura de archivo de NetWare 4 empieza con el objeto Volumen .
Un objeto Volumen es un tipo de objeto Hoja.
Se refiere a un volumen físico en el que se almacenan los archivos y que permite a los Servicios del Directorio NetWare ubicarlos.

El directorio raíz &lsqb;Root&rsqb; de un volumen físico también está representado como parte de un objeto Volumen.
Este es el nivel más alto de la estructura del sistema de archivo.
El objeto Volumen proporciona un enlace entre los Servicios del Directorio NetWare y el sistema de archivo de cada volumen físico.

Los directorios y archivos no son objetos en el árbol del Directorio; son sólo una parte del sistema de archivo.
Para acceder a los directorios y archivos del árbol del Directorio se abre un objeto Volumen o se configuran los objetos Asignación de directorio.

Para que les sea más fácil a los usuarios encontrar los archivos, puede definir objetos Asignación de directorio que apunten a una ubicación concreta en la estructura de sistema de archivo de un volumen.

Usuarios de la red

Sólo podrá acceder a una red de NetWare si el sistema operativo NetWare le identifica como un usuario de la red.

Usuarios de NetWare 2 y NetWare 3

NetWare 2 y NetWare 3 permiten cuatro tipos de usuarios de red, que tienen diferentes niveles de responsabilidad en la misma:

Los usuarios regulares de la red son los que trabajan en ésta.
Pueden ejecutar aplicaciones y trabajar con archivos según los derechos que tengan asignados.

Los operadores son usuarios regulares de la red que tienen asignados privilegios adicionales.
Por ejemplo, un operador de la consola del servidor de archivos es un usuario de la red que tiene derechos específicos para usar la utilidad FCONSOLE.

Los gestores son usuarios a los que se ha dado la responsabilidad para crear o gestionar otros usuarios. 
Los gestores de grupos de trabajo pueden crear y gestionar usuarios; los gestores de cuentas de usuario pueden gestionar pero no crear usuarios.
Los gestores funcionan como supervisores de un grupo concreto, pero no tienen equivalencia de supervisor.

Los supervisores de la red son los responsables del funcionamiento de toda la red.
Mantienen el sistema, reestructurándolo y actualizándolo según las necesidades.

Usuarios de NetWare 4

Los Servicios del Directorio NetWare identifican los mismos usuarios de la red que NetWare 2 y 3.
Además, NetWare 4 identifica un objeto Usuario llamado ADMIN.

La primera vez que el supervisor de la red entra, lo hace como objeto Usuario ADMIN, que se crea por defecto al instalar NetWare 4.

El término supervisor de la red en NetWare 4 se refiere a la persona responsable de configurar la red.
Con NetWare 4 varios usuarios pueden tener privilegios de supervisión- a diferencia del supervisor único requerido para NetWare 2 y 3.

Cuando se crea el usuario ADMIN en el primer servidor de NetWare 4 instalado, éste puede gestionar todos los objetos porque tiene derecho de Supervisión sobre éstos.
Al principio, ADMIN puede gestionar también los directorios y archivos de todos los volúmenes del árbol del Directorio.

El usuario ADMIN no tiene ningún significado especial como tenía el usuario SUPERVISOR en versiones anteriores de NetWare.
Es el primer objeto Usuario creado y, por consiguiente, tiene derechos garantizados para crear y gestionar todos los objetos.

Para utilizar la red, todos los usuarios deben tener un asignado un objeto Usuario en el árbol del Directorio.
El objeto Usuario representa al individuo y hace un seguimiento del uso que esa persona hace de la red.

Entrada

Antes de poder usar cualquiera de los servicios de la red, debe conectarse y entrar en la red.

Los usuarios del Bindery entran o se conectan a servidores individuales.
Es por eso que necesitan una contraseña para cada servidor al que entran.

Los usuarios de los Servicios del Directorio NetWare entran en la red.
Por eso, sólo necesitan una contraseña para acceder a todos los recursos de la red que tienen derecho a usar.

Iniciar y entrar desde una estación de cliente consiste siempre en las mismas tareas:

- Arranque de la estación de trabajo cliente.
Carga automáticamente el sistema operativo del Cliente NetWare cuando lo conecta o reaaranca pulsando ++ .

- Carga del software de red y los controladores.
- Cambio de la unidad por defecto a una unidad de red.
- Entrada con un nombre de usuario y (normalmente) una contraseña.
NetWare le solicita su nombre de usuario y contraseña y, a continuación, completa la conexión a la red.


La forma en que los usuarios ejecutan estos pasos dependerá de cómo el supervisor haya configurado la red y la estación de trabajo cliente.

Los cuatro pasos se pueden ejecutar automáticamente de manera que lo único que tengan que hacer los usuarios sea escribir las contraseñas.
Para obtener más información, consulte el capítulo 7," Configuración de estaciones clientes para entrar en la red", en la página 189 .

Por ejemplo, los usuarios pueden entrar en la red escribiendo LOGIN ESAYERS Intro o LOGINnombre servidor/ESAYERS Intro 

LOGIN se ejecuta normalmente desde el AUTOEXEC.BAT u otro archivo de procesamiento por lotes. 
Hay algunas diferencias en la sintaxis de la línea de comandos según la versión de NetWare. 

Si usa NetWare 3.12, consulte Guía de referencia de las utilidades de NetWare 3.12 para obtener una descripción completa de la utilidad.
Si utiliza NetWare 4, consulta Guía de referencia de las utilidades de NetWare 4 para obtener una descripción completa de la utilidad.

Cuando los usuarios acceden a los recursos de la red, los procesos de autenticación de segundo plano de NetWare 4 verifican que los usuarios sean quien dicen ser.

Una vez los usuarios han entrado en la red, los derechos de Trustee de los mismos en el Bindery o Directorio les permiten acceder a la red.

- Caché
- Redirigido al servidor a través del protocolo de ráfagas de paquetes
- Redirigido a través de Servicios de archivo (FIO)

Capa o nivel de protocolo de transporte

Esta capa o nivel se encarga del mantenimiento de conexiones, del suministro de transmisiones de paquetes entre conexiones, y de la ejecución de otras funciones específicas del transporte. 

El módulo IPXNCP utiliza el protocolo IPX TM como mecanismo de transporte.

El módulo AUTO.VLM vuelve a conectar una estación cliente a un servidor y restablece el entorno de la estación -sin incluir los elementos específicos de los archivos- a su estado original antes de perder la conexión.

Gestión de comunicación y de conexión

El Requester DOS de NetWare incluye la gestión de comunicación y de conexión, a fin de controlar el software del cliente NetWare. 

Gestor VLM

El gestor VLM (VLM.EXE) controla los envíos de memoria y la comunicación en el software individual de VLM.
Es el responsable de cargar los módulos requeridos y de enviar peticiones a módulos individuales.

Por tanto, en lugar de cargar NETX, EMSNETX, XMSNETX, o BNETX, el usuario carga únicamente el programa VLM.EXE.

El gestor VLM decide si un programa VLM dado utiliza memoria expandida, memoria extendida, memoria convencional o bien cualquier otro tipo de memoria soportada, sin afectar a los programas VLM individuales.

Gestor de tablas de conexión

El Gestor de tablas de conexión (CONN.VLM) expande todos los niveles de la arquitectura del Requester DOS de NetWare. 

CONN permite que estaciones clientes ejecuten el Requester DOS de NetWare a fin de establecer un número configurable de conexiones con varios servidores de NetWare.

Compatibilidad inversa con aplicaciones NETX

El programa NETX.VLM proporciona compatibilidad inversa con NETX y otras versiones anteriores de Shell.
Por tanto, no es preciso ejecutar NETX u otras versiones de Shell con el Requester DOS de NetWare.

No obstante, dado que el Requester DOS de NetWare es un redirector en el postprocesamiento de DOS, funciona de forma distinta que el software del Shell de NetWare bajo circunstancias específicas.

Uso de la memoria

El Requester DOS de NetWare trabaja con memoria extendida, memoria expandida y memoria convencional.
En el caso de la memoria extendida, el Requester se carga en la memoria LIM XMS 2.0 y no en el Area de memoria alta (HMA), lo cual significa que la memoria extendida puede coexistir con DOS en carga alta.

El programa VLM.EXE selecciona automáticamente el mejor uso posible: la memoria extendida en primer lugar, la memoria expandida en segundo lugar, y la memoria convencional sólo en caso de ausencia de las opciones de memoria ampliada.

Protocolos y transportes de NetWare

El software de NetWare que permite que las estaciones clientes se comuniquen y se comprendan en la red se divide en dos componentes, que consisten en un protocolo , el cual gestiona datos, y un transporte , el cual gestiona los mensajes de aplicación.
Ello puede proporcionarse en una o varias piezas de software.

Para poder comunicarse en una red, las estaciones clientes deben utilizar un protocolo idéntico al que se está utilizando en la red.

En algunos casos, las estaciones se encuentran configuradas para utilizar varios protocolos.
En tal caso, las estaciones clientes pueden comunicarse con redes distintas simultáneamente. 

Por ejemplo, una estación que utilice los protocolos IPX y TCP/IP únicamente podrá comunicarse con servidores configurados por los protocolos IPX o TCP/IP.

Los protocolos del cliente NetWare soportan desde la Capa o nivel 3 (la capa o nivel de red) a la Capa o nivel 4 (la capa o nivel de transporte) del modelo de referencia de red de Interconexión de sistemas abiertos (OSI) de la Organización internacional para la estandarización (ISO).

El software del cliente NetWare proporciona los siguientes protocolos estándar:

- ARP (Protocolo de resolución de direcciones)
- BOOTP (Proporciona información de configuración de TCP/IP) 
- ICMP (Protocolo de control de mensaje internet)
- IP (Protocolo Internet)
- IPX/SPX TM (Intercambio de paquete de interred/Intercambio de paquetes secuenciados TM)
- MIB (Base de información de gestión)
- NetBIOS (Interfaz de servicios NetBIOS)
- RARP (Protocolo de resolución de dirección inversa)
- RPL (Carga de programas remotos)
- SNA (Arquitectura de la red del sistema)
- SNMP (Protocolo simple de gestión de red)
- TCP (Protocolo de control de transmisión)
- UDP (Protocolo de datagramas de usuario)
- XNS (Sistema de red Xerox*)

Si desea más información sobre cómo optimizar las estaciones cliente en el uso de protocolos de transporte de NetWare, consulte el capítulo 2, "Descripción de las opciones de NET.CFG", en la Guía de referencia técnica para el cliente DOS y MS Windows NetWare.

Protocolo IPXODI

Aunque el Requester DOS de NetWare intercepta y prepara peticiones para transmisión de red, la entrega real la realiza el protocolo IPX.
El software del cliente NetWare lo soporta a través de la Interfaz abierta de datos-enlace de intercambio de paquetes (programa IPXODI.COM).

IPXODI añade encabezado a cada paquete de datos.
El encabezado especifica la información necesaria para la entrega de redes destino al anunciar el origen del paquete, su destino, y lo que sucede después de su entrega.

La capa o nivelcidad de destinación de IPXODI es importante, pero no puede garantizar, por sí sola, que la entrega de una transmisión de datos se lleve a cabo con éxito.

IPXODI transmite paquetes de datos como datagramas (paquetes autocontenidos que se mueven independientemente del origen al destino) y, por tanto, puede entregar los paquetes sólo en base a un esfuerzo máximo.
La entrega se garantiza sólo cuando se usa el protocolo SPX.

Dirección de la memoria base.
La dirección de la memoria base especifica la ubicación en la memoria que utiliza la tarjeta de red para intercambiar información a través de la red.
También se conoce como dirección de inicio RAM.
Algunas tarjetas de red no utilizan RAM; si su tarjeta de red pertenece a este tipo de tarjetas, no es necesario que especifique un valor para la definición.
La dirección de memoria base que a menudo está disponible para la tarjeta de red es D800 (a veces escrito como D8000). 
Si utiliza un gestor de memoria (como por ejemplo el EMM386 o el QEMM*), es posible que necesite excluir esta dirección de memoria base en caso de que el gestor de memoria no sea capaz de detectar automáticamente que la tarjeta de red la está usando. 

Tipo de trama de medios.
Todas las estaciones de trabajo clientes y servidores que usen una única dirección de red deben utilizar el mismo tipo de trama.
Si ya posee una conexión de red, lea el parámetro del tipo de trama para la opción LINK DRIVER del archivo NET.CFG. 
En NetWare 2 y NetWare 3 TM, los controladores LAN Ethernet de Novell tienen por defecto el tipo de trama de Ethernet 802.3.
En NetWare 4 TM, el tipo de trama por defecto de los controladores ODI de la LAN de Ethernet es el 802.2.
Si se utiliza un tipo de trama Ethernet 802.2 en estaciones de trabajo clientes, dichas estaciones de trabajo no podrán conectarse a una red que espere el tipo de trama Ethernet 802.3.
A fin de evitar un conflicto potencial, sitúe ambos tipos de trama (Ethernet 802.2 y Ethernet 802.3) en el mismo orden en el servidor y en estaciones de trabajo clientes.
Si desea más información acerca de los tipos de trama que reciben soporte de controladores de tarjeta de red comunes, consulte "tipo de trama" bajo la "Opción Link Driver" en el capítulo 2 de la Guía de referencia técnica para cliente DOS y MS Windows NetWare.

Introducción

El software de NetWare para estaciones clientes remotas incluye el soporte para arranque remoto y las tramas de Carga de programa remoto (RPL) de la arquitectura de IBM, tal como se define en el manual IBM Remote Program Load User's Guide .

El arranque remoto se basa en el concepto de almacenamiento de la imagen de un disquete arrancable de un volumen de NetWare.
Las estaciones de trabajo de arranque remoto usan esta imagen para arrancar y conectarse a un servidor.

La tecnología RPL se basa en el mismo concepto de almacenamiento de un archivo imagen de un disquete arrancable de un volumen de NetWare.
Sin embargo, RPL depende de un programa de carga que se descarga en todas las estaciones de trabajo remotas para su conexión a un servidor.

- Utilice más de un archivo imagen de disquete por dirección de nodo.

- Software del programa LAN Support Program de IBM.

- Compatibilidad con controladores ODI TM de la LAN y con DOS 5.0 y versiones posteriores

- Soporte del arranque de estaciones clientes remotas en un bridge de encaminamiento de origen.


Estas estaciones clientes no necesitan una unidad de disquete duro ni de disquete para funcionar en la red y, por lo tanto, se denominan estaciones de trabajo sin disquete .

Una estación de trabajo sin disquete se basa en un chip de Memoria programable de sólo lectura (PROM) instalado en la tarjeta de la red para poderse comunicar con el servidor de arranque. 

Cuando se enciende la estación de trabajo, ésta usa la imagen de arranque almacenada en un servidor para cargar el software del sistema DOS y los archivos de cliente NetWare TM necesarios que se usan para la conexión a la red.

El archivo imagen puede incluir los archivos que cargaría normalmente desde un disquete de arranque antes de cargar el software del Requester DOS de NetWare TM (el archivo VLM.EXE). 

NetWare le permite usar un archivo imagen por defecto para todas las estaciones de trabajo sin disquete de la red, o utilizar un solo archivo imagen para cada sistema y entorno de red concreto de la estación de trabajo.

Para configurar el arranque remoto o el RLP en estaciones clientes de NetWare se requiere la configuración de un servidor de NetWare o de un servidor de RPL, la configuración del hardware en cada estación de trabajo cliente de arranque remoto, y la configuración del software de una estación de trabajo cliente con una conexión al servidor de NetWare desde el que desee que se arranquen las estaciones de trabajo remotas.

La configuración completa para una estación cliente de arranque remoto requiere que complete las siguientes tareas: 

- Realización de un disquete de archivo imagen de arranque 
- Configuración del soporte de RPL
- Creación de archivos imagen de disquete de arranque remoto 
- Arranque de estaciones de trabajo remotas

Instalación de un chip PROM certificado de Novell de arranque remoto

Para la instalación de memoria PROM de arranque remoto certificada por Novell debe completar las siguientes tareas: 

- Inserte una memoria PROM de arranque remoto en cada estación de trabajo que desee arrancar desde la red.

- Determine el tipo de memoria PROM de arranque remoto que está usando.


Instalación de un chip de memoria PROM de arranque remoto certificado por Novell en la tarjeta de red

Para obtener información sobre cómo completar este procedimiento, consulte la documentación proporcionada con el kit de la memoria PROM de arranque remoto certificado por Novell.

Determinación del tipo de memoria PROM de arranque remoto 

Existen tres clases de memoria PROM de arranque remoto que se usan con redes de NetWare.
Se hace referencia a estos tipos de la siguiente forma:

Antiguo (PROM de arranque remoto IPX TM de Novell).
En la inicialización, el PROM de arranque remoto IPX de Novell envía una trama de protocolo GET NEAREST SERVER a través del cable.
Esta trama es una identificación indudable de que se trata de una memoria PROM de arranque remoto de IPX de Novell. 

Este tipo de memoria PROM de arranque remoto soporta el arranque remoto y no requiere la configuración del soporte RPL.
Consulte "Realización de un disquete de archivo imagen de arranque" en la página 166 para continuar.

Ampliada (PROM de arranque remoto ampliada de Novell). 
En la inicialización, el PROM de arranque remoto mejorado de Novell envía una trama de protocolo 802.2 FIN a través del cable.
Esta trama es una identificación indudable de que se trata de una memoria PROM de arranque remoto ampliada que utiliza un protocolo 802.2 FIND-FOUND para conectarse a un servidor y descargar un archivo de carga.

No utilice el archivo RPLODI.COM con este tipo de PROM. 
El RBOOT.RPL tiene la posibilidad de comunicarse con controladores ODI de la LAN que se carguen desde archivos imagen de arranque. 

Este tipo de memoria PROM de arranque remoto soporta la RPL y requiere la configuración del soporte RPL.
Consulte "Configuración del soporte de RPL" en la página 159 para continuar.

IBM RPL PROM (PROM de programa remota de IBM).

En la inicialización, el PROM de RPL de IBM envía una trama de protocolo 802.2 FIN a través del cable.
Esta trama es una identificación indudable de que la memoria PROM utiliza un protocolo 802.2 FIND-FOUND para conectarse a un servidor y descargar un archivo de carga.

No utilice el archivo RPLODI.COM con esta PROM de RPL de IBM.
La posibilidad de comunicarse con controladores ODI de la LAN que se carguen desde archivos de carga se proporciona en los siguientes archivos: TOKEN.RPL, ETHER.RPL, PCN2L.RPL, F1ETH.RPL, RBOOT.RPL.

Este tipo de memoria PROM de arranque remoto soporta la RPL y requiere la configuración del soporte RPL.
Consulte "Configuración del soporte de RPL" en la página 159 para continuar.

El tipo antiguo de las memorias PROM de arranque remoto sólo soporta los tipos de tramas originales 802.3 y E-II.
El soporte de tipo mejorado sólo soporta el tipo de trama 802.2.

Por lo tanto, es importante conocer qué tipos de trama soporta el servidor para determinar si se soporta el tipo de PROM de arranque remoto que está usando.

Si el servidor soporta el tipo de trama E-II de Ethernet, puede usar entonces cualquier tipo de PROM de arranque remoto. 

Algunos procedimientos para configurar el arranque remoto en la estación de trabajo cliente son específicos para el tipo de PROM de arranque remoto que esté utilizando.
Para obtener información sobre el tipo de PROM de reinicio remoto, consulte la documentación del fabricante.

Configuración del soporte de RPL

La configuración RPL de la red sólo es necesaria para estaciones de trabajo remotas que utilicen las memorias PROM ampliadas de Novell o de RPL de IBM.
Las estaciones de trabajo remotas que utilicen las PROM "antiguas" o de arranque remoto de IPX de Novell no necesitan configurar el soporte de RPL.
Consulte "Realización de un disquete de archivo imagen de arranque" en la página 166 para obtener más información.

Para configurar el soporte a RPL debe realizar las siguientes acciones:

Cargue el programa RPL.NLM en el servidor de NetWare y asócielo con el controlador LAN del servidor.

Cargue el archivo RPL.COM en una estación de trabajo cliente si necesita arrancar desde un servidor de NetWare a través de un bridge de encaminamiento de origen.

Asegúrese de que los archivos necesarios se copian en los directorios apropiados.

Copie los archivos RPL nuevos en los directorios apropiados si los archivos existentes no son actuales.

Todos los archivos necesarios para configurar una estación de trabajo cliente remota de NetWare se encuentran en el directorio RPL del disquete 2 de Cliente NetWare para DOS y MS Windows.

Carga de programas RPL.NLM

El archivo RPL.NLM es un programa del Módulo cargable de NetWare TM (NLM) que actúa como un stack de protocolo.
Responde a las tramas de Carga de programa remoto (RPL) de la arquitectura de IBM tal como se define en el manual IBM Remote Program Load User's Guide .

El programa RPL.NLM se usa en redes que tienen instaladas estaciones de trabajo sin disquete con el módulo RPL BIOS, que se conocen como PROM de arranque remoto . Actualmente, se da soporte a RPL en las tarjetas de red siguientes:

- Tarjetas Ethernet de IBM (MCA y Modelo 25SX)
- Tarjetas de red PC de IBM
- Tarjetas de red Token-Ring de IBM
- Tarjetas de Novell (o compatibles) y tarjetas de red de otros fabricantes que tengan instalada la memoria PROM de arranque remoto mejorada

Procedimiento

En la consola del servidor, cargue el programa RPL.NLM escribiendo: LOAD RPL Intro 

Asocie el programa RPL.NLM a la tarjeta de red del servidor escribiendo: BIND RPL TOtarjeta_de_red&lsqb;parámetro&rsqb; Intro 

Por ejemplo, para asociar RPL a un controlador LAN NE2000 con el tipo de trama Ethernet_802.2, debe escribir: BIND RPL TO NE2000 FRAME=ETHERNET_802.2 Intro 

Para obtener detalles sobre cómo utilizar el parámetro BIND para RPL en la línea de comandos, consulte "Parámetros BIND de RPL", en el Capítulo 3 de la Guía de referencia técnica para el cliente DOS y MS Windows de NetWare .

Cuando se carga el programa RPL.NLM, éste busca el archivo BOOTCONF.SYS en el directorio SYS:LOGIN.
Si encuentra uno, el programa RPL.NLM lee el archivo BOOTCONF.SYS en memoria para que la información de archivo esté disponible cuando se reciba una trama FIND de una estación de trabajo cliente.
Consulte "Configuración del archivo BOOTCONF.SYS" en la página 179 para obtener más información. 

Carga del archivo RPL.COM

El archivo RPL.COM es un programa Residente en la memoria (TSR) cargado en una estación de trabajo cliente.
Responde a las tramas de Carga de programa remoto (RPL) de la arquitectura de IBM tal como se define en el manual IBM Remote Program Load User's Guide .

El archivo RPL.COM permite que una estación de trabajo cliente funcione como servidor de RPL para la descarga del archivo de arranque en estaciones de trabajo remotas.

Se usa en redes que tienen estaciones de trabajo remotas sin disquete que necesitan conectarse a un servidor de NetWare mediante un bridge de encaminamiento de origen con el módulo RPL BIOS, conocido como PROM de arranque remoto .
Actualmente, se da soporte a RPL en las tarjetas de red siguientes: 

- Tarjetas Ethernet de IBM (MCA y Modelo 25SX)
- Tarjetas de red PC de IBM
- Tarjetas de red Token-Ring de IBM
- Tarjetas de Novell (o compatibles) y tarjetas de red de otros fabricantes que tengan instalada la memoria PROM de arranque remoto mejorada

Procedimiento

Asegúrese de que se ha instalado memoria PROM de arranque remoto ampliada de Novell en la tarjeta de red de la estación de trabajo cliente que desea configurar como un servidor de RPL. 

Cree un directorio LOGIN en la raíz de la estación de trabajo cliente escribiendo: MD LOGIN Intro 

Expanda o copie el archivo RPL.COM y los programas de carga en el directorio LOGIN.

Consulte "Copia de los nuevos archivos RPL y de arranque del servidor" en la página 165 para obtener más información. 

En una estación de trabajo cliente dentro del mismo segmento de la LAN que las estaciones de trabajo remotas, cargue el archivo RPL.COM escribiendo: RPL Intro 

Asocie el programa RPL.COM a la tarjeta de red añadiendo la línea siguiente al archivo NET.CFG: RPL de protocolo bindtarjeta de red &lsqb;parámetro&rsqb; 

Por ejemplo, para asociar RPL a un controlador LAN NE2000 con el tipo de trama Ethernet_802.2, debe escribir: RPL de protocolo bind ne2000 frame ethernet_802.2 

Debe definir PS= nombre de servidoro el el parámetro opcional de BIND de GNS para que dirija las estaciones de trabajo remotas al servidor de NetWare que contiene el archivo imagen del disquete de arranque para cada estación de trabajo.

Consulte "RPL de protocolo" en "Opción RPL de protocolo " en el capítulo 2 de Guía de referencia técnica para el cliente DOS y MS Windows NetWare para obtener más información y generar una lista de los parámetros BIND opcionales para RPL.

Introducción

La configuración correcta de la estación de trabajo cliente puede hacer que la entrada en la red sea virtualmente transparente.
Este capítulo supone que ya se han realizado las siguientes tareas: 

- Se ha creado un objeto Usuario o cuenta de usuario para cada usuario

- Está instalado el software del Cliente NetWare TM 

- Se ha creado un directorio personal para cada usuario
Los usuarios con estaciones clientes NetWare 2 ó 3 conectadas a una red NetWare 4 deben tener creado un objeto Usuario en el contenedor donde está definido el contexto del servidor basado en el Bindery (los servicios del Bindery están definidos por defecto en todos los servidores NetWare 4 instalados).
Los usuarios de NetWare 2 ó 3 no necesitan conocer su contexto porque entran en el servidor en vez del árbol de Directorio.


Modificación del archivo NET.CFG para la entrada en NetWare 2 y NetWare 3

Cuando entra en una red de NetWare 2 o NetWare 3 TM , se conecta a un servidor individual.
Entonces puede realizar conexiones adicionales a otros servidores en los que tenga una cuenta de usuario.

Uso del parámetro Preferred Server

Utilice el parámetro Preferred Server en el archivo NET.CFG para dirigir el software de cliente al servidor donde está ubicada su información de usuario.

En redes con varios servidores, este parámetro es muy útil para agilizar la entrada en la red.
Este parámetro le permite también entrar sólo con su nombre de usuario.

Indique el parámetro como se muestra a continuación:

preferred server = nombre_servidor 

No use los parámetros Preferred Server y Preferred Tree simultáneamente.
El segundo parámetro prevalecerá sobre el primero. 

Adición del parámetro Preferred Server al archivo NET.CFG 

Abra el archivo NET.CFG con un editor de texto.

Su archivo NET.CFG de cliente se copió en el directorio de destino para archivos de DOS especificado durante la instalación del software.

En el encabezamiento de Requester DOS de NetWare TM escriba: preferred server=nombre_servidor .

Por ejemplo, escriba: preferred server=mktventas 

Ejemplo de la sección Requester DOS de NetWare

NetWare DOS Requester Preferred Server=mktventas First Network Drive=f 

Modificación del archivo NET.CFG para la entrada en NetWare 4

Cuando entra en NetWare 4 TM no se conecta a servidores individuales, como en versiones anteriores de NetWare, sino que entra en la red.

Con el nombre completo (DN), puede simplificar el proceso de entrada añadiendo información al archivo NET.CFG de la estación de trabajo cliente.
Para hacerlo, debe añadir líneas bajo el encabezado NetWare Dos Requester en el archivo NET.CFG.

Si añade líneas para los siguientes parámetros, no será necesario que escriba su nombre completo cada vez que entra en la red después de arrancar su estación de trabajo cliente:

- Name Context
- Preferred Tree

Debe conocer su nombre de contexto antes de realizar estas modificaciones, si no no puede añadir entradas para los parámetros.
Para obtener información acerca de cómo configurar el nombre del contexto de la estación de trabajo cliente consulte "Comprensión de la entrada a los Servicios del Directorio NetWare", en la página 194 .

Uso del parámetro Name Context

Al usar el parámetro Name Context (nombre del contexto) en el archivo NET.CFG, informa al Requester DOS de NetWare de su nombre del contexto en el árbol del Directorio.
La utilidad LOGIN puede ubicar su objeto Usuario y conectarlo a la red.

Indique el parámetro como se muestra a continuación: name context="nombre_contexto" .

La definición por defecto de este parámetro se encuentra en la raíz, lo que puede causar confusión si hay nombres de usuario duplicados en un árbol del Directorio.

Si usa una versión anterior del shell de NetWare en su estación de trabajo cliente, el parámetro Name Context no es válido.
NetWare 4 crea un objeto Usuario para las estaciones clientes que usan los archivos del shell de NetWare en el mismo contenedor que el objeto Servidor NetWare en el que va a entrar.
Por defecto, los servicios del Bindery están definidos para todos los objetos Servidor NetWare que están instalados.
Por lo tanto, los usuarios entran directamente al servidor en vez de entrar al árbol del Directorio.

Uso del parámetro Preferred Tree

Al usar el parámetro Preferred Tree en el archivo NET.CFG, el Requester DOS de NetWare se dirige al árbol de Directorio en el que está definido su nombre de contexto.
De esta manera, la utilidad LOGIN puede ganar tiempo al buscar sólo el árbol del Directorio preferente.

Añada este parámetro a su archivo NET.CFG si está usando NetWare 4 en una red con varios árboles del Directorio.

Indique el parámetro como se muestra a continuación: preferred tree=nombre_árbol .

La definición por defecto de este parámetro se encuentra en la raíz.
Por eso, es importante que este parámetro se utilice para evitar cualquier confusión si hay nombres de usuario duplicados en árboles de Directorio separados.

No use los parámetros Preferred Server y Preferred Tree simultáneamente.
El segundo parámetro prevalecerá sobre el primero. 

Modificación de la opción Requester DOS de NetWare

La opción Requester DOS de NetWare le permite configurar la manera en que el software de su estación de trabajo cliente interactúa con la red.

Para modificar esta sección de opción en el archivo NET.CFG, siga los siguientes pasos:

Procedimiento

Abra el archivo NET.CFG con un editor de texto.

El archivo NET.CFG está ubicado en el directorio de su estación de trabajo cliente, que es el directorio de destino especificado durante la instalación del software del cliente. 

En el encabezamiento de Requester DOS de NetWare, escriba: name context="nombre_completo" 

Por ejemplo, escriba: name context="esayers.ventas.novell us" 

(Condicional) Si su red tiene varios árboles del Directorio, escriba preferred tree=nombre_árbol_preferente 

Por ejemplo, escriba: preferred tree=novell us 

Ejemplo de la sección Requester DOS de NetWare

NetWare DOS Requester name context "Esayers.ventas.novell us" preferred tree=novell us first network drive=f 

Comprensión de la entrada a los Servicios del Directorio NetWare

La arquitectura de los Servicios del Directorio NetWare TM (NDS) facilita la entrada en la red.

La información de usuario reside en una base de datos global.
Se asigna una posición a cada usuario, o contexto , en la base de datos global que informa a la red de la ubicación del objeto Usuario del usuario en relación a la red entera.

En el archivo NET.CFG, se hace referencia a su contexto Usuario como nombre del contexto .

Se le da al objeto Usuario un nombre denominado nombre común .
Cuando entra en la red, la utilidad LOGIN debe saber dónde está ubicado el objeto Usuario particular---el contexto.

El contexto del objeto Usuario forma también el Nombre completo .
La vía de acceso desde el objeto a la raíz del árbol constituye el Nombre completo del objeto, que es exclusivo---distinto a los Nombres completos de los demás Nombres completos de objetos.

El Nombre completo de su objeto Usuario es la suma de los Nombres del árbol de abajo a arriba en diagonal, desde el objeto hasta la raíz.

Uso del formato correcto del Nombre completo para entrar en la red

Cuando utiliza un Nombre completo de un objeto Usuario, se lista primero el nombre común, seguido de un punto (.); luego el nombre de la unidad organizativa (un objeto contenedor), seguido también de un punto; y así hasta el nombre del objeto Organización (y el nombre del objeto País, si se usa).

De esta manera, el Nombre completo del objeto Usuario podría estar representado por:

- Nombre común.
- Nombre de unidad organizativa.
- Nombre de objeto Organización.
- Nombre de objeto País

La estructura real del Nombre completo dependerá de cómo esté definida la red.

Especificación del tipo de nombre de un objeto

Un tipo de nombre distingue la clase del objeto al que hace referencia, como por ejemplo objeto Usuario u objeto Unidad organizativa.

Por ejemplo, podría expresar ESAYERS.VENTAS.NOVELL US de la siguiente manera: CN=ESAYERS.OU=VENTAS.O=NOVELL US donde CN es el nombre común del objeto Usuario, OU es el nombre de la unidad organizativa, y O es el nombre de la organización. 

Cuando se desplaza de un objeto contenedor a otro, debe incluir siempre el tipo de nombre de un objeto al escribir el Nombre completo o el contexto de éste.

Debe incluir siempre los tipos de nombre de los objetos en Nombres completos o contextos al incluir el objeto País en el árbol del Directorio.
Por lo tanto, aunque haga referencia a objetos ubicados en el mismo objeto contenedor, deberá designar los tipos de nombre (CN, OU, o O).

Modificación del contexto

Cuando se desplaza de un objeto contenedor a otro, cambia contextos .
Siempre que cambie los contextos, es posible que deba indicar el Nombre completo del objeto al que cambia el contexto.

Por ejemplo, en la siguiente figura, si el objeto Usuario Brent ubicado en Test.MedTec desea obtener información del objeto Servidor llamado SERVER1 que está ubicado en el contexto Development.MedTec, Brent deberá referirse al objeto Servidor al cambiar el contexto para acceder a SERVER1.

Cuando haya entrado en la red, utilice las siguiente utilidades para cambiar el contexto del objeto Usuario en el árbol del Directorio: 

- La utilidad CX de la línea de comandos de DOS 
- La opción "Cambiar contexto", en la utilidad NETUSER para DOS
- La ventana "Conexiones de la unidad" en las Herramientas del usuario de NetWare para MS Windows 

Entrada en su contexto

Escriba LOGINcontexto Intro 

Sustituya contextopor su Nombre completo.
Por ejemplo, escriba: LOGIN ESAYERS.VENTAS.NOVELL US Intro 

Introduzca la contraseña de usuario (si es necesario). 

Dónde ir desde aquí

Para obtener más detalles, consulte el capítulo 1, "Comprensión de los Servicios del Directorio NetWare" en Introducción a los Servicios del Directorio NetWare.

Introducción

El procedimiento de entrada en un servidor o red de NetWare proporciona, a la estación de trabajo cliente, acceso a los recursos de la red como, por ejemplo, los directorios, impresoras y aplicaciones. 

Cuando se inicia una petición de entrada, el sistema operativo NetWare consulta los derechos de seguridad específicos del usuario y a continuación le solicita que introduzca una contraseña.

Toda la información de seguridad se colocará en la lista de conexiones del servidor de NetWare y la estación de trabajo estará conectada.

En este punto, la utilidad LOGIN ejecuta uno o más guiones de entrada (con lo que se inicializan las variables de entorno, se asignan las unidades de red, etc.).

Con las modificaciones necesarias realizadas en el archivo NET.CFG, sólo necesitará el nombre de usuario y la contraseña para entrar en el servidor que desee para las redes NetWare 2 y NetWare 3 TM o bien en el contexto del nombre y el árbol que desee para las redes NetWare 4 TM .

Requisitos previos

Antes de entrar, compruebe lo siguiente:

- El software de Cliente NetWare TM está cargado.

- Si la estación de trabajo cliente puede utilizar un controlador de tarjeta de red Interfaz abierta de enlace de datos TM (ODI), cargue el software siguiente: LSL.COM El controlador de tarjeta de red (por ejemplo, NE2000.COM) IPXODI.COM VLM.EXE.

- Si el cliente utiliza exclusivamente un controlador IPX TM dedicado (no un controlador ODI TM ), cargue los programas siguientes: IPX.COM VLM.EXE

- (Condicional: si utiliza MS Windows) El software de Cliente NetWare se ha cargado antes de cargar MS Windows.

- (Aconsejable) El archivo NET.CFG se ha modificado para la entrada.

- Si no modifica el archivo NET.CFG, se requiere el Nombre completo para cada entrada.
Consulte el Capítulo 7, "Configuración de estaciones clientes para entrar en la red," en la página 189 para obtener más información.

Introducción

NetWare le permite configurar y gestionar las unidades de la red en todas las estaciones clientes.
De esta forma, las estaciones clientes pueden almacenar y acceder a archivos y aplicaciones que se encuentran en un sistema de archivo central.

La capacidad de almacenar y acceder a estos archivos y aplicaciones es posible a través de las asignaciones de unidad que especifican las ubicaciones concretas en los volúmenes y los directorios de la red.

Las asignaciones de unidad permiten navegar con mayor facilidad por la estructura del directorio y acceder a los discos duros de los servidores de NetWare y otros medios de almacenamiento, tales como los dispositivos de CD-ROM y unidades de cinta.

Hay tres tipos de asignaciones de unidad en un entorno de NetWare:

- Unidades locales Especifican los discos duros instalados en una estación de trabajo cliente.
DOS reserva las cinco primeras letras (A-E) para asignaciones de unidad.

- Unidades de red Especifican volúmenes o directorios concretos de la red.
La cantidad de unidades de la red disponibles dependerá de la definición del parámetro LASTDRIVE de DOS en el archivo CONFIG.SYS.
En versiones anteriores de software para cliente de NetWare, la primera unidad de red era siempre la primera letra de unidad no reservada al uso local.
Este ajuste se hacía con el parámetro LASTDRIVE de DOS en el archivo CONFIG.SYS.


Es necesario añadir el parámetro LASTDRIVE=Z al archivo CONFIG.SYS, ya que el software Requester DOS de NetWare es un redirector del postprocesamiento de DOS.
La primera unidad de la red ahora está definida en el archivo NET.CFG.

Consulte "FIRST NETWORK DRIVE" en "Opción Requester DOS de NetWare", en el Capítulo 2 de Guía de referencia técnica para el cliente DOS y MS Windows NetWare, para obtener más información al respecto.

El Requester DOS de NetWare utiliza la primera unidad de DOS disponible después del parámetro FIRST NETWORK DRIVE definido en el archivo NET.CFG.
Por consiguiente, las estaciones de trabajo cliente que accedieron, por ejemplo, a una definición de parámetro "LASTRIVE=G" en su CONFIG.SYS se utilizan para ver la unidad H: como la primera unidad de la red en vez de la primera letra de unidad después de la última letra de unidad local.
Para mantener la compatibilidad con los archivos de procesamiento por lotes existentes, posiblemente deseará añadir el parámetro "FIRST NETWORK DRIVE = H" debajo de la opción Requester DOS de NetWare del archivo NET.CFG.
Si no, deberá alterar los archivos de procesamiento por lotes que asuman la unidad H: como primera unidad.

Cómo establecer asignaciones de unidad en un orden concreto 

En muchos casos, puede serle útil asignar las unidades de la red en el orden siguiente:

Asigne la primera unidad de red al directorio local del usuario.

Si los usuarios ejecutan MS Windows desde la red, asigne una unidad al directorio de cada usuario que contenga archivos específicos del usuario.

Asigne las unidades restantes a los directorios en los que los usuarios trabajan frecuentemente, como los directorios de proyectos o las aplicaciones de red.

Cómo establecer asignaciones de unidad para entornos diversos. 

En vez de especificar letras de unidad como H o G, puede serle útil usar un asterisco seguido de un númeronpara representar lanª unidad de la red.

Por ejemplo, si la primera unidad de la red es F: y usa xxx , a dicha unidad se le asigna la letra H: (ya que MAP xxx etc.).
Si la primera unidad de la red, en cambio, corresponde a la letra D: y usa "MAP *4:=" , a dicha unidad se le asignará la letra G:.

De esta forma las letras de unidad se reordenan automáticamente cuando se eliminan o añaden unidades locales o cuando se cambia la primera unidad de la red.

Asimismo, ello permite que los usuarios puedan entrar desde estaciones de trabajo con un número de unidad local distinto a su estación regular.

Puede asignar una unidad local (generalmente de A: a C:) a un directorio de red, pero no podrá acceder a la unidad local hasta que elimine la asignación de unidad de la red.

No es posible asignar una unidad redirigida, como una unidad de CD-ROM, a una unidad de red.

Asignación de unidades de búsqueda

Puede añadir volúmenes o directorios de red a la vía de acceso de DOS usando la opción de unidad SEARCH en la utilidad MAP.
Esto es útil para las aplicaciones y archivos de red a los que desea acceder, sea cual sea el directorio actual en el que está trabajando.

Se permite un máximo de 16 unidades de búsqueda de NetWare. 

Al asignar una unidad de búsqueda, utilice un número de unidad de búsqueda (una S seguida de un número).
Este número establece la siguiente letra de unidad disponible para la asignación, desde la Z hacia atrás en el alfabeto inglés.

La letra se pone en la instrucción de DOS.
Si ya tiene unidades en la instrucción PATH, MAP S1 sobreescribirá la primera de la vía de acceso.

Para impedir que las asignaciones de unidad prevalezcan sobre las letras PATH de DOS existentes, use la opción INSERT al asignarlas.
Por ejemplo, escriba"MAP INS S16:= vía ".

Para asegurarse de que puede acceder a las utilidades de NetWare, a los directorios de DOS y a las aplicaciones, debería asignar unidades de búsqueda en el orden siguiente :

Asigne la primera unidad de búsqueda (S1:) al directorio SYS:PUBLIC, que contiene las utilidades de NetWare para las estaciones clientes de DOS y MS Windows.

Asigne la segunda unidad de búsqueda (S2:) al directorio DOS si accede a DOS desde la red.

Asigne la tercera unidad de disco y las siguiente (S3:, S4:, etc.) a los directorios que contienen aplicaciones de la red, archivos de proyectos y directorios de datos usados con frecuencia.

Si ejecuta MS Windows desde la red, asigne una unidad de búsqueda al directorio de MS Windows para el grupo MS Windows. 

Para evitar cambiar sin darse cuenta el nombre de una unidad de destino que tiene que estar asignada a una letra de unidad concreta, puede asignar todas las unidades de búsqueda restantes con el número S16:, o usar INSERT para insertar una asignación de unidad.

Si tiene una aplicación que requiere una letra de unidad concreta, puede usar el comando siguiente para asignar la unidad de búsqueda: MAP S16:=unidad:=vía_acceso 

Si asigna una unidad de búsqueda usando un número que ya se ha asignado, NetWare convierte la unidad de búsqueda antigua en una unidad de red.

Asignación de unidades a los objetos Asignación de directorio (sólo NetWare 4)

Otra forma de asignar una unidad a un directorio es crear un objeto Asignación de directorio que lo especifique.
Asi mueve el directorio, sólo necesita cambiar el objeto Asignación de directorio en vez de todas sus asignaciones.

Por ejemplo, para asignar una unidad de búsqueda a un objeto Asignación de directorio cuyo nombre completo es VENTAS_LA.NOVELL_US, escriba MAP S2:=.VENTAS_LA.NOVELL_US Intro 

En el ejemplo anterior, el nombre del objeto Asignación de directorio empieza con un punto, lo que indica que la unidad está asignada a la unidad raíz.

Si su objeto Usuario existe en el mismo contexto que el objeto Asignación de directorio, como en el ejemplo VENTAS_LA.NOVELL_US, el comando MAP no tiene que especificar el nombre completo del objeto Asignación de directorio.

En caso contrario, introduzca la siguiente información MAP S2:=APPL Intro .

En su contexto actual es recomendable utilizar objetos Asignación.
No utilice nombres completos que indiquen otros contextos. 
Si su asignación está en otro contexto, debería crear un alias que especifique el objeto Asignación de directorio real.
Consulte "Objeto alias" en Conceptos para obtener más información. 

Consulte el Capítulo 3, "Creación de guiones de entrada ", en Supervisión de la red para obtener más información sobre los objetos Asignación de directorio.

Introducción

El software de soporte de conmutación de tareas para NetWare, los archivos TBMI2.COM y TASKID.COM actúan como buffer y gestor entre las peticiones de IPX TM y SPX TM y un proceso de llamada de aplicación durante la conmutación de tareas.

Cuando se realiza una conmutación de tareas durante un proceso de llamada de una aplicación, el software de soporte de conmutación de tareas de NetWare asegura que los protocolos de transporte IPX y SPX envíen la llamada al recurso de la red adecuado.
De esta manera, se evitan rupturas de comunicaciones entre IPX y los procesos de software cuando se realiza una conmutación de tareas.

Puede necesitar este soporte si usa un conmutador de tareas de DOS o si conmuta sesiones de DOS en MS Windows en modo real o estándar.
Si la aplicación no necesita el software de soporte de conmutación de tarea de NetWare y no lo usa, la estación de trabajo cliente podría experimentar fallos de sesión o podría colgarse.

Si la única aplicación IPX/SPX TM que usa es el Requester DOS de NetWare, no necesitará cargar los archivos de conmutación de tarea.
Si no está seguro de si la aplicación necesita soporte de conmutación de tarea, continúe y ejecute los archivos de conmutación de tarea; sólo usan un pequeña cantidad de memoria.
Después de ejecutar la aplicación, introduzca el parámetro de la línea de comandos /D y observe el número en el campo "Uso de llamada lejana".
Si este número es cero, la aplicación no ha usado los archivos de conmutación de tarea y, por lo tanto, puede ejecutarla sin estos archivos.

Deberá usar los archivos de soporte de conmutación de tarea de NetWare si el entorno presenta cualquiera de las características siguientes:

Desea conmutar entre aplicaciones en DOS o en sesiones de DOS en MS Windows

La aplicación atraviesa el software del Requester DOS de NetWare y accede a IPX o SPX directamente

Está ejecutando MS Windows en modo real o estándar

Introducción

El módulo de soporte ODI TM NDIS (ODINSUP) es un stack de protocolo de Interfaz abierta de enlace de datos TM (ODI) que realiza la interfaz entre NDIS y los controladores LAN de Cliente NetWare®.

ODINSUP convierte las transmisiones del NDIS de aplicaciones basadas en NDIS en un formulario que se ajusta a los controladores ODI.

ODINSUP traduce también las transmisiones recibidas desde la red en un formato que el Gestor de protocolos NDIS (PROTMAN) entiende.

NDIS especifica dos tipos de controladores: uno para la manipulación del hardware, y otro para el tratamiento del protocolo.
Los controladores relacionados con el hardware se denominan también controladores MAC (Control de acceso a medios) de NDIS.
Los controladores de tratamiento del protocolo se denominan controladores PROTMAN de NDIS.

Al separar estas funciones, NDIS le permite tener varios protocolos con el mismo adaptador físico, o usar sólo un controlador por adaptador físico.
El Gestor de protocolos NDIS regula las transacciones entre los controladores de hardware y de protocolo. 

ODINSUP proporciona una interfaz MAC de NDIS para cada adaptador físico del sistema.
ODINSUP puede estar asociado con una opción NET.CFG a un máximo de cuatro adaptadores y puede generar una interfaz MAC separada y distinta para cada uno.
Ello equivale a tener cargados los controladores que se ajustan a NDIS correspondientes y una interfaz ODI completa.

ODINSUP proporciona las siguientes ventajas:

No es necesario configurar ni arrancar de nuevo ningún controlador para pasar de una red basada en NDIS a una red de NetWare.

Compatibilidad entre aplicaciones escritas para NDIS y los controladores que se ajustan a ODIS.

Asociación de los controladores ODI de la LAN al stack de protocolo NDIS

El archivo NDIS PROTOCOL.INI es necesario para indicar a los protocolos NDIS a qué MAC deben asociarse y los que deben utilizar.

Normalmente, se puede eliminar toda la información de PROTOCOL.INI para controladores MAC de NDIS, si sólo se utiliza un controlador MAC de NDIS y un stack de protocolo.
No se necesita ninguna información concreta de ODINSUP en el archivo PROTOCOL.INI.

Para asociar los controladores ODI de la LAN al stack de protocolo NDIS, deberá editar el archivo PROTOCOL.INI con un editor de texto para:

- Asociar el stack de protocolo de NDIS a los controladores ODI de la LAN
- Eliminar la asociación en línea del stack de protocolo de NDIS a los controladores MAC MAC de NDIS

Procedimiento

Abra el archivo PROTOCOL.INI con un editor de texto ASCII. 

Por ejemplo, para usar el editor de sistema de Novell®DOS TM 7 en la línea de comandos y editar el archivo PROTOCOL.INI en el directorio BOOT, introduzca la siguiente información EDIT C:&bsol;BOOT&bsol;PROTOCOL.INI Intro 

Localice en las líneas todos los casos que asocian los controladores MAC de NDIS.

Puede buscar Bindigns =controlador_MAC_NDIS 

Por ejemplo, para buscar un controlador de NDIS para la tarjeta 3Com* EtherLink II* , busque la siguiente línea: Bindings = elkinii.dos 

Si no sabe el nombre del controlador de NDIS que busca, consulte la documentación incluida en su tarjeta de red.

Convierta en comentarios (con un punto y coma) todas las líneas de asociación que haya encontrado.

Por ejemplo, el archivo PROTOCOL.INI para un controlador 3Com* EtherLink II LAN puede aparecer de la siguiente manera: &lsqb;ETHERNET&rsqb; . .; Bindings = elkinii.dos 

Después de cada línea de asociación inhabilitada con un comentario, añada una línea para asociar el protocolo NDIS al controlador ODI de la LAN.

Siga la misma sintaxis que la línea que inhabilita, usando el nombre del controlador ODI de la LAN en vez del nombre del controlador de NDIS.

Por ejemplo, para añadir una línea para 3C503.COM del controlador 3Com* EtherLink II ODI de la LAN 3C503.COM, escriba: &lsqb;ETHERNET&rsqb; . .; Bindings = elkinii.dos Bindings = x3c503 

Puesto que los nombres de controladores en PROTOCOL.INI no pueden empezar con un número, escriba una x antes de los controladores 3Com o demás controladores que empiezan con un número (Por ejemplo: Bindings = x3C503 ). 

(Condicional) Si tiene dos o más tarjetas de la red usando el mismo controlador ODI de la LAN, escriba un número de ejemplo para asociar el protocolo NDIS a una aparición concreta de la tarjeta.

Por defecto, el protocolo NDIS utiliza la primera tarjeta de este tipo.

Para que NDIS use una tarjeta que no sea la primera encontrada, puede especificar un número de ejemplo.

Escriba el número de ejemplo al final del nombre del controlador, sin ningún espacio entre el nombre del controlador y el número de ejemplo.

Por ejemplo, si tiene dos tarjetas de red, escriba un número de ejemplo para la segunda tarjeta de manera que NDIS use ésta, como se muestra a continuación: &lsqb;ETHERNET&rsqb; . .; Bindings = elkinii.dos Bindings = ne20002 

(Condicional) Si dispone de controladores ODI adicionales, asocie a los mismos el protocolo NDIS.

Para asociar el protocolo NDIS a varios controladores, escriba el nombre de éstos en la misma línea y separados por una coma.

Por ejemplo, para asociarlo al controlador NE2000 y al NE1000, escriba Bindings = ne2000,ne1000 

Guarde y salga del archivo PROTOCOL.INI.

Introducción

NetBIOS (Sistema de entrada y salida básica de red) es un protocolo estándar de IBM para aplicaciones creadas con el fin de ejecutar comunicaciones de par a par en la red IBM PC y en la red Token-Ring de IBM.
NetBIOS ha llegado a tener una aceptación general como estándar para interfaces de red.

El Cliente TM NetWare® para DOS y MS Windows proporciona un controlador NetBIOS que emula el protocolo NetBIOS.
Este emulador permite al IPX TM de NetWare enlazar con la interrupción NetBIOS 5Ch y una interfaz alterna, Interrupción 2Ah.

El NetBIOS suministrado por Novell es un emulador porque no transmite paquetes NetBIOS.
En su lugar, los paquetes NetBIOS se encapsulan en paquetes IPX y éstos a su vez se transmiten. 

El protocolo NetBIOS fue diseñado para redes de pequeña escala.
Mientras Novell ha añadido funcionalidad a la especificación original, la emulación NetBIOS funciona aún con mayor eficacia con redes de pequeña escala.

Por otra parte, si su red contiene varios miles de estaciones de trabajo y sus segmentos LAN se encuentran interconectados con más de una ruta, use SPX TM en lugar de NetBIOS si sus aplicaciones soportan SPX.

El soporte de Cliente NetWare para el emulador NetBIOS se debe configurar en cada estación de trabajo de cliente desde el que desea acceder a las aplicaciones NetBIOS.

La configuración completa para estaciones de trabajo cliente que utilizan NetBIOS exige que se completen las siguientes tareas: 

- Instalación de una aplicación NetBIOS
- Instalación de los archivos de Cliente NetWare y NETBIOS.EXE

Introducción

Los conductos nombrados conforman la base de la comunicación entre el cliente y las aplicaciones avanzadas de cliente-servidor, como por ejemplo el software del Servidor SQL Microsoft y del Comm Server Microsoft.

Los procesos de cliente-servidor proporcionan un método de proceso informático de red distribuido en el que dos estaciones de trabajo cliente cooperan en la ejecución de una aplicación. 

El componente cliente de la aplicación (el preprocesamiento) funciona en una estación de trabajo cliente, mientras que el componente servidor (el postprocesamiento) funciona en una segunda estación de trabajo.

Los procesos informáticos de cliente-servidor permiten un uso más efectivo de los recursos de cálculo, un rendimiento más alto, mayor flexibilidad, actualizaciones más simples y, (en el caso de ciertas aplicaciones), una mayor fiabilidad e integridad de datos.

En el caso de las estaciones clientes de NetWare®, la comunicación entre aplicaciones de cliente-servidor se implementan en su mayoría mediante el Interprocess communication protocol (IPC) de conductos nombrados.

Puede configurarse una estación de trabajo cliente de conductos nombrados para operar en modo por defecto o par.

En el modo por defecto, las estaciones de trabajo clientes de conductos nombrados obtienen nombres para los servidores de conductos nombrados mediante la petición de una tabla de router (o compatible) IPX TM.
En este modo, las estaciones clientes de conductos nombrados solicitan nombres de servidor específicos según se necesite.

En el modo par, las estaciones de trabajo clientes de conductos nombrados conservan su propia tabla de servidor de conductos nombrados.
Cuando se carga un extensor de conductos nombrados en modo par, se emite una Consulta general de servicios y se compilaran las respuestas de los servidores de conductos nombrados en una tabla en cada estación de trabajo cliente.

Para configurar en modo par, consulte "NP MAX MACHINE NAMES número" debajo de "Opción Named Pipes" en el Capítulo 2 de la Guía de referencia técnica para el cliente DOS y MS Windows NetWare , si desea obtener más información.

El soporte de Cliente NetWare para el protocolo de conductos nombrados (el archivo DOSNP.EXE) debe configurarse en cada estación de trabajo cliente a la que se desee acceder desde las aplicaciones cliente-servidor de conductos nombrados.

En la mayoría de los casos, deberán definirse las opciones de configuración del software de la Link Support Layer TM (LSL), la capa de soporte de enlace.

La configuración completa de estaciones de trabajo clientes que utilizan el protocolo de conductos nombrados requiere que se realicen las siguientes tareas:

- Instalación de un servidor de conductos nombrados y una aplicación de cliente-servidor
- Instalación del extensor de conductos nombrados y archivos de Cliente NetWare
- Modificación de los archivos de configuración de Cliente NetWare y de Sistema DOS

Introducción

El software del Agente de servicio de destino (TSA), (el archivo TSASMS.COM) es un programa que procesa datos que se trasladan entre un destino específico y las aplicaciones de los Servicios de gestión de almacenamiento (SMS) como la utilidad SBACKUP o el programa Navegador de NetWare.

Una aplicación basada en la arquitectura SMS que se ejecuta en el servidor, envía peticiones al software de TSA.
El software de TSA realiza las siguientes funciones:

- Recibe comandos desde la aplicación SMS y los traduce a peticiones de datos para que pueda manejarlos el sistema operativo local.

- Transmite las peticiones de datos al sistema operativo local.

- Recupera los datos pedidos desde el sistema operativo local y los devuelve a la aplicación SMS en formato SMS estándar.


EL software de TSA de Novell tiene las características siguientes:

- Opera como un programa residente en memoria (TSR)

- Ocupa menos de 7 KB de memoria (usando los ajustes por defecto)

- Soporta MS DOS (3.3 y posterior), DR DOS® (5 y posterior) y Novell DOS TM (7 y posterior).

- Proporciona varios niveles de seguridad.

- Proporciona mejorar el tamaño ajustable de los buffers de lectura para rendimiento.

- Proporciona tamaño ajustable del stack para reducir el uso de memoria general.


El soporte de Cliente NetWare para el software TSA (el archivo TSASMS.COM) debe estar configurado en cada estación de trabajo cliente que desea que se gestione mediante una aplicación SMS.

La configuración completa para las estaciones clientes con el software TSA requiere que se realicen las siguientes tareas: 

- Instalación de una aplicación SMS

Si la zona horaria no está definida en el entorno de su estación de trabajo, es posible que los mensajes de alerta para SNMP indiquen una hora incorrecta.

Guarde los cambios y salga.

Modificación del archivo NET.CFG

Las modificaciones siguientes del archivo NET.CFG son necesarias para cada estación de trabajo cliente en la que desee cargar software del Escritorio SNMP.

- Añada las líneas necesarias en el archivo NET.CFG para cargar y soportar el software de escritorio SNMP.

- Defina la dirección del "DESTINO DE MENSAJES DE ALERTA" de las consolas de Gestión de Servicios de NetWare (NSM) u otros gestores basados en SNMP.


Consulte "Archivo NET.CFG de muestra para servicios de Escritorio SNMP", en la página 338 , para obtener un ejemplo. 

Por defecto, los servicios de Escritorio SNMP no están habilitados cuando instala el software Cliente NetWare.
Para habilitar servicios de Escritorio SNMP, realice el siguiente procedimiento:

Procedimiento

Asegúrese de que los archivos de Escritorio SNMP estén copiados en el directorio de Cliente NetWare (el directorio por defecto es C:&bsol;NWCLIENT).

Modifique el archivo NET.CFG para cargar los archivos .VLM de escritorio SNMP.

Para cargar la serie de archivos SNMP .VLM, modifique el archivo NET.CFG en su estación de trabajo de cliente en cualquier de los modos siguientes:

- Para cargar el software de Escritorio SNMP y Requester DOS de NetWare, añada las siguientes líneas al archivo NET.CFG en el orden proporcionado : netware dos requester vlm = wssnmp.vlm vlm = wstrap.vlm vlm = wsreg.vlm vlm = wsasn1.vlm vlm = mib2if.vlm vlm = wsdrvprn.vlm vlm = mib2prot.vlm .

- Para cargar sólo el Escritorio SNMP, sin los módulos del Requester DOS de NetWare, añada las siguientes líneas al archivo NET.CFG en el orden proporcionado : netware dos requester use defaults = off vlm = wssnmp.vlm vlm = wstrap.vlm vlm = wsreg.vlm vlm = wsasn1.vlm vlm = mib2if.vlm vlm = wsdrvprn.vlm vlm = mib2prot.vlm .


Consulte las siguientes secciones en el capítulo 2, "Opción de Requester DOS de NetWare", de la Guía de referencia técnica para el cliente DOS y MS Windows NetWare para obtener detalles sobre cómo configurar el Escritorio SNMP: 

- "Tiempo límite asíncrono
- "Nombres de comunidad"
- "Soporte MIB-II"
- "Proveedores de transporte"
